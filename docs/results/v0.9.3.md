The dlmalloc malloc-limit (`memoryLimit=16MB`, re-applied on `QuickJS.create` AND `QuickJS.restore`) catches native-C allocations as catchable OOM **before** `memory.grow` can WS-1006 the DO. Live, one session, 5 bomb shapes:

- 258MB `Uint8Array(...).fill` → `NativeAllocLimitError`, alive, next eval = 101

After all bombs: `gen` reported `inMemory:true`, `generation:1` — **the DO never reconstructed, never WS-1006'd.** Local node proof: 258MB alloc throws with the WASM buffer never growing past the 1.2MB baseline.

Also added: typed `NativeAllocLimitError` + `_formatError` relabel of "out of memory" (distinct from the v0.8 bytecode `MemoryLimitError`); pre-flight `HOST_ARG_MAX_BYTES=8MB` checks on inbound `__hostCall` arg blob and outbound tool result (`host.ctx.slice/get` already capped at 1MB).

## (2) Engine-migration journal — PASS
New `cell_journal` SQLite table: each committed eval cell's source appended after checkpoint; effectful cells flagged (host.fetch/subLM/kv/tools/Math.random/Date/crypto). On engine-hash mismatch at restore, `ensure_glue` routes to `glue.replayJournal()` (fresh engine + re-hydrate kv/ctx + re-eval journaled sources in order), instead of a hard `EngineHashMismatchError`.

Test: 4-cell pure namespace (`acc=65`, `fib` closure), `journalLen:4`. Evicted, fired test-only `{t:engineBump}` to force `snapshotEngineHash != engineHash`, cold-eval'd:
- `restoreSource: "journal-replay"` (not the old hard error)
- recovered `acc=65`; replayed closure `fib(12)=144`
- effectful-cell session (`Math.random`) also recovered via journal-replay (best-effort path exercised)
- **Normal engine-match cold restore still uses the heap-snapshot fast path `sqlite-restore` (NOT journal)** — confirmed; journal only fires on real mismatch.

Honesty caveat (ADR-0002 by design): `result.faithful=true` only for pure-only journals. Effectful cells (network/kv/entropy) cannot re-fire faithfully and are flagged. Effectful flagging is a conservative host-side substring scan (no JS parser in the Rust DO) — may over-flag, affecting only the honesty flag, not recovery. Journal is per-cell SOURCE, not a general replay engine; heap-snapshot remains the durability mechanism.

## Operating-envelope notes
- 16MB live-heap ceiling = `MAX_CELL_USED_BYTES` (below the 18MB dump ceiling, above the ~7MB stdlib envelope). Legitimate working sets >16MB must chunk across cells (documented, unchanged).
- `{t:engineBump}` is a TEST-ONLY hook gated behind a message type; never fires in normal operation. A real engine upgrade is detected by the build-time SHA-256 of `quickjs.wasm` changing.
- v12 runs the v1.1-parity kernel (engineHash `v11-...`), which predates the v0.9.3 relabel — its native-bomb errors surface as `InternalError`/`MemoryLimitError` but still catch cleanly with the facet alive (isolation intact). The relabel + journal are v093-specific and verified on montydyn-v093.
