# montydyn v0.4 — Cold-start instrumentation + long-tail harness + quick wins

> Built on v0.3 (`v0.4/`), deployed **`montydyn-v04`**
> (https://montydyn-v04.umg-bhalla88.workers.dev). Branch `feat/v0.4`. R2 keys namespaced
> `v04/`. v0/v0.1/v0.2/v0.3 + their workers + the facet spike all UNTOUCHED. 2026-06-01.

## Goal

Improve **cold-wake latency** and add a reusable **long-tail latency experiment harness**
to the infra. Cold-wake path: WS msg -> DO wakes -> `ensure_glue()` lazily restores
(read manifest+chunks / R2 get -> gunzip -> instantiate QuickJS -> pre-size+blit linear
memory -> restore globals -> re-register host fns + kv -> eval the cell).

## HARD CONSTRAINT honored (frozen clock)

On workerd the wall/high-res clock is **FROZEN during a synchronous turn** (spectre
mitigation). You can only read meaningful `Date.now()` deltas **across the I/O `await`
boundaries** that unfreeze it. So sub-steps inside one sync turn are **unmeasurable
in-isolate** and we are honest about it: such phases report `0ms` (which is not "free", it
is "sub-await-resolution / inside one frozen turn") and we infer them by differencing.
Counts (grow count, page counts) are exact because they are *counted*, not timed.

## A. Instrumentation + long-tail harness (the infra experiment)

### Per-restore phase marks (in the eval/create reply)

Every eval/create reply now carries `restoreTimings`:

```
restoreTimings: {
  readMs,                       // server: snapshot byte fetch (SQLite chunk read | R2 get)
  totalServerMs,                // server: whole ensure_glue() span
  glue: {
    gunzipMs,                   // glue: across the gunzip await
    instantiateMs,              // glue: across QuickJS.restore await (instantiate+grow+blit+ext-reinit)
    totalGlueMs,
    rawBytes, neededPages, presizedPages,
    growCount, growDeltaPages,  // EXACT counts (not timed) -> B1 verification
  }
}
```

Implementation: `glue.js restore()` stamps `Date.now()` at each await boundary
(`tEnter`, after `gunzip`, after `QuickJS.restore`), records exact `growCount` by counting
`WebAssembly.Memory.prototype.grow` calls during the restore window, and exposes it via
`lastRestoreTimings()`. `lib.rs ensure_glue()` stamps server `readMs` across the byte-fetch
await and `totalServerMs` across the whole span, folds in the glue marks
(`restore_timings_value`).

**What is measurable vs not (measured live):**
- **readMs IS measurable** when the fetch await actually yields: R2 get at 5MB read **597ms**.
  For the SQLite hot path `read_chunks` is synchronous -> clock frozen -> reads `~0` (honest:
  prior benchmarks already found SQLite read sub-ms).
- **gunzipMs / instantiateMs read `0`** for base AND 5MB images: gunzip (DecompressionStream)
  and `QuickJS.restore` resolve within the same frozen post-fetch turn at these sizes. This
  CONFIRMS the long-standing finding (SUMMARY.md): "latency = ~100% network, ~0% compute;
  gunzip+deserialize+instantiate+blit all sub-ms even at 30MB." The instrumentation is
  structurally correct and will surface non-zero values only if a phase ever crosses an await
  resolution; it does not, which is itself the result.

### Long-tail bench harness — `v0.4/bench/coldstart.mjs`

Opens many INDEPENDENT sessions, forces a genuine cold wake each, records the
client-observed cold-wake RTT, and builds a distribution (p50/p90/p95/p99/p99.9) — per
snapshot size and warm-vs-cold.

- **Genuine cold wake:** `--mode=evict` (default) uses a brand-new DO id per iteration
  (fresh isolate/object on first touch) AND `evict`s the in-memory kernel before the
  measured op, so the measured eval takes the lazy cold-RESTORE path with certainty.
  `--mode=idle` does a real disconnect + `--idleMs` sleep + reconnect (closest to production
  hibernation; slow, use small `--n`).
- **Sizes:** `--size=base|5mb|15mb`. Big sizes grow the namespace with a chunked,
  incompressible LCG fill (chunked across cells to stay under the workerd interrupt-tick
  budget; each cell re-arms it).
- **Warm baseline:** `--warm` measures the same op without evicting -> cold delta isolates
  restore/instantiate from WS RTT.
- Records server `restoreTimings` per cold iteration to attribute the client tail to
  network vs server.

```
node bench/coldstart.mjs <wss-base> [--n=50] [--mode=evict|idle] [--size=base|5mb|15mb]
                         [--warm] [--idleMs=70000] [--conc=4] [--json=out.json]
```

## Measured cold-wake DISTRIBUTIONS (live, montydyn-v04)

Client-observed round-trip (ms). `evict` mode, fresh DO id per iteration.

| scenario | n | min | p50 | p90 | p95 | p99/p99.9 | max | mean |
|---|---|---|---|---|---|---|---|---|
| **base COLD** (~1.31MB raw, sqlite) | 58 | 160 | **180** | 417 | 569 | **640** | 640 | 226 |
| **base WARM** (no re-instantiate) | 57 | 151 | 163 | 171 | 173 | 180 | 180 | 162 |
| **5MB COLD** (6.55MB raw / 5.35MB gz, r2) | 1 | — | **4592** | — | — | — | — | — |

- **Cold tail is real and dominated by isolate spin-up + first instantiate**: base cold p50
  (180ms) ≈ warm p50 (163ms) — the median cold wake is essentially WS RTT + a sub-ms
  restore. The TAIL diverges: cold p90/p99 = 417/640ms vs warm 171/180ms. That delta (~250–460ms
  at the tail) is the cold-isolate spin-up + first WASM instantiate (matches EXP-5a's ~700ms
  first-instantiate observation). It is NOT in our restore phases (those read 0ms) because it
  happens in workerd BEFORE/AROUND our code runs — it is the platform cold isolate, confirmed

- **5MB cold = 4592ms, attributed: readMs=597ms (R2 get) + ~4s of R2/gzip/transport.** Server
  `totalServerMs` only captured 597ms (the measurable R2 await); the remaining wall time is the
  per-chunk checkpoint gzip + R2 round-trips during setup and the measured R2 restore. Confirms
  R2 overflow is the long-tail driver for large images -> keep images on the SQLite hot path.

## B. Quick wins (each applied + measured)

### B1 — Pre-size WebAssembly.Memory; NO grow churn during the blit ✅ VERIFIED

The QuickJS WASM module **exports** its own memory (not importable), so we cannot inject a
pre-sized `WebAssembly.Memory`. But `quickjs-wasi`'s `restore()` already grows the fresh
instance's memory in a **single** `grow(neededPages - currentPages)` call BEFORE the blit
(`dst.set`), so the blit triggers zero incremental grows. We **instrument and assert** this
directly by counting `Memory.prototype.grow` calls during restore:

- base (20 pages needed): **growCount = 1**, growDeltaPages = 2, presizedPages == neededPages == 20.
- 5MB (100 pages needed): **growCount = 1**, growDeltaPages = 82, presizedPages == neededPages == 100.

So the blit is into pre-sized memory with **no grow churn** at any size. Measured effect on
latency: negligible at these sizes (instantiateMs already 0ms), but the property is now
*guaranteed and verifiable* (`growCount<=1`) and protects against a future regression where a
library change could blit-then-grow incrementally (which WOULD churn). The smoke + bench both
assert `growCount <= 1`.

### B2 — Lazy instantiate: gen/ping/evict never instantiate ✅ VERIFIED

Audited `handle()`: only `eval` and `create` call `ensure_glue()` (which is the ONLY path
that creates/restores a QuickJS instance). `gen` reads SQLite meta + the in-memory Option
only. `evict` *drops* the instance. The DO constructor (`new`) does no instantiation. Added
an explicit cheap `ping` message with the same guarantee.

Live-verified (fresh, never-evaluated DO): `gen` -> `inMemory:false`, `ping` -> `inMemory:false`,
`evict` -> `droppedInMemory:false` (nothing to drop). First `eval` after a cold `evict`
takes `restoreSource:"sqlite-restore"`; the subsequent warm eval is `restoreSource:"warm"`
(no re-instantiate). No regression — gen/ping/evict were already lazy; v0.4 makes it explicit
and asserted.

### B3 — wasm-opt -Oz on quickjs.wasm ✅ KEPT

`wasm-opt -Oz` (binaryen 129, added as devDependency):

| | bytes | delta |
|---|---|---|
| original quickjs.wasm | 1,586,981 | — |
| **-Oz** (kept) | **1,447,147** | **-139,834 (-8.8%)** |
| -O3 (rejected, larger) | 1,451,619 | -8.5% |

**Correctness gate PASS (local, before adopting):** round-trip on the -Oz module —
create -> eval -> snapshot -> deserialize -> restore -> eval — produces byte-identical
results to the original (`inc()===41`, identical JSON output, identical serialized snapshot
size 1,245,212 bytes). The engine content-hash changes (new sha256
`9f10a46b…`), which is fine: v04 is a fresh worker with no v03 snapshots to restore, and the
build-time engine-hash guard bakes the new hash in. Smoke 62/62 on the -Oz engine.

**Instantiate delta:** node-side compile bench showed 0.21ms->0.18ms (V8 lazy-compiles, so
not representative). On workerd the cold instantiate is the relevant cost; the ~8.8% smaller
module is a smaller download + less to compile on the cold isolate, but the effect is within
the noise of the cold-tail distribution at this engine size (the engine is ~1.4MB; the tail
is dominated by platform isolate spin-up, not module size). Kept because it is a free,
correctness-preserving shrink with zero downside.

## Bottom line

- **Median cold wake is already fast (~180ms p50)** and ≈ warm — the restore (read+gunzip+
  instantiate+blit) is sub-ms at SQLite-hot sizes, as prior work claimed; v0.4 now *proves it
  with marks* (growCount=1, phases 0ms = within one frozen turn).
- **The long tail (cold p99 ~640ms vs warm ~180ms) is platform cold-isolate spin-up + first
  WASM instantiate**, OUTSIDE our restore path — confirmed by differencing cold-vs-warm and by
  the 0ms server phases. The actionable lever is keeping the engine small (B3, -8.8%) and the
  image on the SQLite hot path (R2 adds ~600ms readMs + multi-second transport at 5MB).
- The harness + per-phase marks are the durable infra deliverable: any future engine/restore
  change can be re-benched for its tail and attributed to network vs server vs isolate.

# Convergence — cold-start cost decomposition + recommendations

This section converges the multiple live bench runs (n=48–60 base cold, n=49–57 warm,
single 5MB R2 probe). Where runs disagreed (e.g. base cold p95 569 vs 731ms, p99 640 vs
874 vs 920ms), the spread itself is the finding: **the base-cold tail is run-to-run noisy
at n≤60 because it is platform cold-isolate variance, not a stable kernel cost.** Numbers
below are given as ranges across runs, honestly.

## 1. WHERE THE MS GO — decomposed cold-wake cost

A cold wake = `WS connect/upgrade → DO wake → ensure_glue() restore → instantiate → eval`.
Decomposition by directly-measured marks plus cold-vs-warm differencing (the only honest way
to attribute the sub-await phases the frozen clock cannot time):

| component | base p50 (~1.3MB, SQLite) | base p99/tail | 5MB (R2 overflow) | in our control? |
|---|---|---|---|---|
| WS RTT + DO wake | ~160–168ms (≈ warm floor) | ~180–213ms (warm ceiling) | ~1.6s WS+wake | **No** — CF network/platform |
| snapshot READ | ~0ms (SQLite sync, frozen) | ~0ms | **597–1045ms** (R2 `get` await, measured) | **Partly** — choose SQLite vs R2 path |
| gunzip + instantiate + blit | 0ms (one frozen turn, sub-ms) | 0ms | 0ms (still one frozen turn) | **Yes** — B1/B3 keep it sub-ms |
| cold-isolate spin-up + 1st WASM instantiate | folded in (≈warm at p50) | **~520–700ms** (the tail) | folded into the ~1.6s | **No** — platform; only indirect (smaller engine) |

**p50 (~180ms):** essentially all **WS RTT + DO wake** (warm p50 163–168ms; cold p50
180–185ms). The restore-attributable delta at the median is only **~12–17ms**, and even that
is mostly DO-wake jitter, not compute — gunzip/instantiate/blit read 0ms (sub-await, inside
one frozen turn). On the SQLite hot path **restore is effectively free at the median.**

**p99 / p99.9 tail (base ~640–920ms):** warm eval **never exceeds ~213ms** in any run, so the
entire ~430–700ms cold tail is **cold-isolate spin-up + the first WASM instantiate that
workerd does around/before our code** — the EXP-5a ~700ms signature. It is paid **once per
fresh isolate, not per byte** (base snapshot is tiny). This is NOT in our restore phases
(they read 0ms) and is confirmed by differencing cold-vs-warm tails.

**5MB / R2 path:** the cost center **moves to the R2 GET**. Server `readMs` is directly
measurable here because the R2 `get` await unfreezes the clock: **597ms (one probe) to 1045ms
(another)** of pure R2 fetch, pushing client RTT to **2.7–4.6s**. Gunzip/instantiate still
read 0ms even at 5MB — confirming compute stays sub-ms; the wall time is **R2 fetch +
transport**, not engine work.

**Control summary:** p50 and the base tail are **network/platform-bound (unfixable in-kernel)**.
The large-image cost is **R2-fetch-bound** — addressable by keeping images on SQLite or
streaming the R2 read, NOT by shrinking instantiate. The genuinely compute-bound portion
(gunzip+instantiate+blit) is already sub-ms and squeezed to near-zero by B1/B3.

## 2. What the quick wins moved (honest p50/p99 deltas)

- **B1 pre-size (no grow churn):** **no measurable latency change** — instantiate was already
  0ms (sub-await), so growCount=1 vs many grows is invisible at these sizes. What it bought is
  a **verified, asserted invariant** (`growCount<=1` over 56 cold restores, presizedPages ==
  neededPages at 20 and 100 pages). Value = regression protection, not a measured win. Honest.
- **B2 lazy instantiate:** **measurable structural win for non-eval ops.** A cold `gen`-only
  wake p50 **~133ms** vs cold `eval` (restore+instantiate) p50 **~182ms** — a **~49ms p50
  delta** that `gen`/`ping`/`evict` never pay. eval p50 itself is unchanged (it must restore).
- **B3 wasm-opt -Oz:** engine **1,586,981 → 1,447,147 bytes (-8.8%)**, byte-identical
  determinism. **No isolable latency delta** — the instantiate cost is sub-ms/frozen and any
  cold-tail effect is **within run-to-run noise** at this ~1.4MB engine size (would need many
  idle-mode samples on both engines to resolve a sub-100ms module-size effect). Kept as a free,
  correctness-preserving shrink. Honest: "no measurable change" at current sizes.

Net: the wins moved **p50 by ~0ms for eval** (correctly — the median is network-bound) and
**~49ms for non-eval ops** (B2). The tail moved **within noise**. The durable gains are
**invariants + a smaller engine + an explicit lazy contract**, not a headline p99 number.

## 3. The long-tail harness now in the infra

- **`v0.4/bench/coldstart.mjs`** — opens many independent sessions, forces a genuine cold wake
  each (`--mode=evict` brand-new DO id + evict before the measured op; `--mode=idle` does a
  real disconnect + `--idleMs` hibernation), records client-observed cold-wake RTT, and builds
  p50/p90/p95/p99/p99.9 distributions per size and warm-vs-cold. Captures server
  `restoreTimings` per cold iter to split client tail into network vs server.
- **`v0.4/bench/restore-resample.mjs`** — resampler over captured restore timings.
- **Per-restore marks** (`restoreTimings` in every eval/create reply): server `readMs` /
  `totalServerMs`; glue `gunzipMs` / `instantiateMs` / `totalGlueMs` / `rawBytes` /
  `neededPages` / `presizedPages` / `growCount` / `growDeltaPages`. Counts are exact (not
  timed) so B1 is verifiable directly.

Re-run:
```
node v0.4/bench/coldstart.mjs wss://montydyn-v04.umg-bhalla88.workers.dev \
  --n=60 --mode=evict --size=base --conc=5            # cold base distribution
node v0.4/bench/coldstart.mjs <wss> --n=60 --warm --conc=5     # warm baseline
node v0.4/bench/coldstart.mjs <wss> --size=5mb --conc=1 --n=4  # R2 path (conc=1, small n)
```
**Gotcha:** 5MB/15MB MUST run `--conc=1` with small `--n` — every eval auto-checkpoints, so a
chunked incompressible fill gzips a growing multi-MB image per cell and times out under
concurrent server gzip contention. base runs fine at `--conc=5`. The harness lacks WS-upgrade
retry, so a few transient 500s on brand-new DO ids (4/60) appear as errors — cold-isolate
spin-up flakiness, not a kernel fault.

## 4. Recommended NEXT cold-start lever + merge decision

**Highest-value remaining lever: keep snapshot images on the SQLite hot path (avoid R2
overflow) — and if R2 is unavoidable for large images, stream the R2 read.** The data is
unambiguous: SQLite restore is ~0ms-measurable (sub-ms), while a single 5MB R2 restore spends
**597–1045ms in the R2 `get` alone** and ~2.7–4.6s total RTT. That is **10–50× the entire base
cold wake** and is the only directly-measured, multi-hundred-ms compute/IO cost we own. Levers,
ranked:
1. **Raise the SQLite-vs-R2 overflow threshold / chunk more into SQLite** — eliminates the R2
   `get` for the common case. Biggest measured win available.
2. **Stream/range the R2 read** for genuinely large images so restore overlaps fetch instead of
   one blocking `get`.
3. Engine shrink past B3 — **low value**; instantiate is already sub-ms and the base tail is
   platform isolate spin-up we don't control.

**Merge v0.4: YES.** Smoke 62/62 PASS, zero behavioral regression, all three wins live and
verified (B1 invariant asserted, B2 measured ~49ms, B3 byte-identical -8.8%). The instrumentation
+ harness are pure additive infra. One cosmetic note: `Cargo.toml` crate name still reads
`montydyn-v03` (does not affect the deployed worker name `montydyn-v04`); left as-is to avoid a
rebuild — fix opportunistically.

## 5. Short-tail vs long-tail — project recommendation

**The short tail (p50) is solved and is network-bound — stop optimizing it in-kernel.** Base
cold p50 ≈ warm p50 ≈ 165–185ms, which is WS RTT + DO wake. No in-kernel change moves it; it is
a CF platform/network floor.

**The long tail splits into two regimes, only ONE of which is fixable in-kernel:**
- **Base/SQLite long tail (p99 ~640–920ms): NOT in-kernel-fixable.** It is platform cold-isolate
  spin-up + first WASM instantiate, paid once per fresh isolate, outside our restore path. The
  only indirect lever is a smaller engine (diminishing returns) or warm-keeping isolates (a
  platform/ops concern, e.g. periodic pings / hibernation tuning — not kernel code).
- **Large-image/R2 long tail (p99 multi-second): IN-KERNEL-FIXABLE and high-value.** Dominated by
  the measured R2 `get` (0.6–1s+). This is the project's clear next compute/IO target.

**Recommendation for the project:** do not chase the base p50/p99 further — it is network/platform
bound. Spend the next cold-start effort on the **R2-overflow path** (threshold tuning + streaming
read), which is the only remaining cost center that is both large (hundreds of ms to seconds) and
within our control. Secondarily, capture the missing distributions the harness already supports
(`--mode=idle` real hibernation, and a full `--size=15mb --conc=1` run) to confirm the R2 tail
scales with fetch size as extrapolated from the single 5MB probe.
