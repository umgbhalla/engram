# montydyn v0 — integrated durable REPL kernel — RESULT

> Branch `feat/v0-kernel`. Built on PATH (b) (Rust DO shell + JS glue), integrating
> EXP-4b (Rust DO + quickjs-wasi), EXP-9 (per-cell checkpoint + engine-hash guard),

## Verdict

**PASS.** The full durable-REPL loop works on real Cloudflare, including a **real idle
eviction** (DO generation 1→2): eval → per-cell SQLite checkpoint → kernel gone →
reconnect → eval restores from SQLite and advances, with **no replay** and **no
re-fired side effects**.

Deployed worker: **`montydyn-v0`** → `https://montydyn-v0.umg-bhalla88.workers.dev`
Version ID at this result: `c8eea08a-cbe3-4ee1-ba84-ef684d154391`.

## What changed vs the experiments

| Concern | EXP behaviour | v0 |
|---|---|---|
| Snapshot store | R2 blob (100% network RTT, ~230ms p50) | **DO SQLite** chunked (~64KB rows), R2 only as >2MB-gz overflow |
| Dump | `serializeSnapshot` + 2nd full copy (~3× transient, EXP-6) | **streaming gzip** (serialize → CompressionStream → single Uint8Array) |
| Engine hash | runtime double-import of wasm bytes (EXP-9) | **baked in at build time** (`scripts/engine-hash.mjs` → `globalThis.__ENGINE_HASH`) |
| Determinism | proven locally (EXP-8) | integrated into the live kernel: seeded clock/RNG/crypto + persisted entropy counters (`clockCalls`,`rngCalls`) per snapshot |
| Size guard | probe-only | in-kernel admission control (refuse >~50MB live / >~20MB raw restore) |
| Concurrency | n/a | in-DO async mutex (JS promise-chain); cell# allocated in critical section |
| Protocol | `cell`/`eval`/`crash` | `eval` (auto-checkpoint), `reset` (clear+epoch++), `gen` |

## Restore latency

`restoreLatencyMs` reported by the DO reads **0** because workerd clamps `Date.now()`
during synchronous compute (Spectre mitigation) — it does not advance across the
restore. So latency was measured **client-side** as the difference between a cold
(restore-triggering) eval round-trip and a warm eval round-trip on the same warm DO:

```
coldRoundTripMs : 155
warmRoundTripMs : 147
restoreCost     : ~8 ms   (SQLite read + gunzip + deserialize + blit)
restoreSource   : sqlite-restore
```

At v0 image sizes (~1.3MB raw / ~103KB gz) the SQLite-local restore cost is **~8ms** —
confirming the EXP-7 finding that decode/instantiate/blit is sub-compute, and the only
reason the R2 path was ~230ms p50 was network RTT. **Moving the hot store to SQLite
removes that RTT.** Total client-observed cold-wake round-trip (~155ms) is dominated by
the client↔edge WebSocket network leg plus DO wake, not by the restore itself.

## Checkpoint-atomicity fix (HIGH-severity, 2026-06-01)

Adversarial review found the snapshot replace in `checkpoint` was **destructive-first
and non-atomic**: it `DELETE`d `snap_chunks` + `snap_manifest` BEFORE writing the new
snapshot, with no enclosing transaction. On the R2 path an `.await` (`bucket.put`) sat
between the DELETEs and the manifest INSERT — a crash/evict in that window destroyed the
prior committed snapshot before the replacement was durable → `read_manifest()`→None →
silent `create_fresh()` → **total namespace loss**. The SQLite hot path had the same gap
(DELETE + N INSERTs + manifest + meta as separate auto-committed statements). And the
chunk-count check was a `debug_assert_eq!`, which **compiles out in release wasm**.

### What changed (`v0/src/lib.rs`)

1. **SQLite hot path — atomic via DO write-coalescing.** workerd **rejects** raw
   `BEGIN`/`COMMIT`/SAVEPOINT against DO SQLite (it errors: *"use state.storage
   transaction() / transactionSync() … interacts correctly with automatic atomic write
   coalescing"*). workers-rs 0.8 `SqlStorage` exposes neither helper. So the fix runs the
   **entire replace** (DELETE old chunks+manifest → INSERT new chunks → INSERT manifest →
   advance `committedCell`) in a **single synchronous DO turn with NO `.await` between the
   statements** (`checkpoint`, ~`lib.rs:438-468`). workerd's output gate buffers all
   writes in a turn and flushes them **atomically**; if the turn throws, the buffer is
   discarded → the prior committed snapshot survives. This is the documented-correct
   mechanism (raw transactions are forbidden precisely because coalescing already does it).
   `committedCell` is advanced in the same turn, preserving commit ordering.

2. **R2 overflow path — swap-then-delete** (`checkpoint`, ~`lib.rs:404-411`, `483-491`;
   new key helper `r2_key_for`, `lib.rs:186-191`). The new object is written under a
   **fresh, cell/epoch-scoped key** (`v0/<doid>/e<epoch>-c<cell>.qjs.gz`) BEFORE the
   manifest commits; the **OLD** R2 object is deleted only AFTER the new manifest is
   durable (post-commit, best-effort). A crash between the `put` and the commit leaves the
   prior committed snapshot (old key + old manifest) fully intact. `reset_critical` now
   deletes the actual committed key from the manifest (plus the legacy fixed key).

3. **Runtime chunk-count guard.** `debug_assert_eq!(blobs.len(), n_chunks)` →
   real runtime check returning a typed `CorruptSnapshotError` (`read_chunks` now returns
   `Result<Uint8Array>`, `lib.rs:~456-489`). A truncated chunk set is now refused instead
   of feeding garbage to gunzip.

No wire-protocol or determinism/engine-hash changes. Mutex/`RefCell` borrow discipline
preserved (no borrow held across `.await`; the SQL block is borrow-free and await-free).

### Re-test evidence (deployed `montydyn-v0`, version `38036fe6-eeaa-46f4-a76f-9af357f2efa4`)

- **SQLite path crash edge (`test-client.mjs sim`): PASS.** checkpoint cell0 → `evict`
  (kernel gone, `inMemory=false`, `committedCell=0`) → restore: `x=42`,
  `restoreSource="sqlite-restore"`, then `inc()=43`. Prior snapshot never lost.
- **R2 overflow path (`test-client.mjs r2`): PASS.** 3.5MB incompressible namespace →
  gz **3.77MB > 2MB SQLITE_HOT_MAX** → `store="r2"`. Two checkpoints produced **fresh
  keys** `…/e0-c0.qjs.gz` → `…/e0-c1.qjs.gz` (swap confirmed). `evict` (crash) → restored
  the 3.67MB namespace from the committed R2 object: `x=43`, `inc()=44`, `big` intact.
- **Authoritative R2 crash-restore.** On a genuinely cold DO (generation 2,
  `inMemoryBefore=false`) a restore read `x=42` straight from the committed R2 overflow
  object via the manifest (`restoreSource="sqlite-restore"`), then re-checkpointed to a
  new key (`e0-c2`) — proving the prior committed object was readable and the swap cycle
  repeats. (Note: the `wrangler r2 object get` CLI reported "key does not exist" for live
  keys due to read consistency/CLI lag; the worker binding reads them correctly — that is
  the load-bearing path.)

### Gotchas discovered

- **DO SQLite forbids raw `BEGIN`/`COMMIT`** — the first fix attempt using them failed at
  runtime with an explicit workerd error. Atomicity must come from synchronous
  write-coalescing (no `.await` mid-replace) or the `state.storage.transactionSync` JS API
  (not bound in workers-rs 0.8). We use coalescing.
- **`wrangler r2 object get` is not a reliable oracle** for just-written DO objects
  (consistency lag); verify via a worker-side restore instead.
- Building a >2MB-gz namespace needs **incompressible** bytes filled by a **pure in-VM
  LCG** — per-byte `Math.random()`/`crypto.getRandomValues` (host-callback crossings) over
  16M elements stalls the DO turn; a 3.5MB LCG-filled buffer is fast and gz-resistant.
