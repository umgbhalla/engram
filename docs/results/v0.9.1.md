# v0.9.1 — Chunked context store (>1MB context survives cold-restore)

Deployed: https://montydyn-v091.umg-bhalla88.workers.dev
Branch: `worktree-wf_d653618d-b12-2` (commit 6bc8ebe)

## Scope

Fixes the HIGH v0.9 bug: a large RLM context stored as a single `ctx_json` TEXT value hit `SQLITE_TOOBIG`, and multi-MB contexts were lost on cold restore. Also fixes the LOW v0.9 bug where `trace` showed `final:null` after reconnect.

## Build summary

- HIGH fix: context store chunked across SQLite `ctx_chunks` (~64KB BLOB rows) + `ctx_n_chunks` manifest column, replacing the single `ctx_json` TEXT value. Restore reassembles via `read_ctx_chunks`; legacy `ctx_n_chunks==0` snapshots fall back to inline `ctx_json` (backward compatible). Chunk writes stay crash-atomic inside the same synchronous DO turn; reset clears `ctx_chunks`.
- `host.ctx.{len,slice,grep,chunk,get}` read across chunks transparently; glue still hydrates into one in-memory Map (no API change).
- Raised single-slice cap `CTX_MAX_SLICE` 256KB -> 1MB (under the 8MB cell-growth tripwire / 18MB dump ceiling).
- Trace fix: persist the RLM trajectory final to SQLite meta (`finalInfo`) on every cell that records one; `{t:final}` reads live glue first, falls back to persisted meta, so `trajectory()` shows the answer after cold reconnect.
- SDK `@montydyn/sdk` bumped to 0.9.1, README updated; NOT npm-published per scope.
- cargo check (wasm32) clean. Reproduced bug on montydyn-v09, confirmed fixed on montydyn-v091.
