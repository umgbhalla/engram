## Feature A — Mid-cell used-heap tripwire (closes the v0.7 Medium gap)

**v0.7 gap (`docs/results/v0.7-guards.md` risk #1):** an in-cell raw-allocation bomb (unbounded
`a.push(...)` loop, a 258MB zod tree, a string-doubling bomb) grows the WASM linear buffer DURING
a single synchronous eval. The post-cell dump guard (`MAX_DUMP_BUFFER_BYTES` 18MB) only runs AFTER
the cell returns, so the cell could hang / lose the in-flight reply before the guard ever fired
(recoverable on reconnect, but degraded).

**Fix (in `_interruptHandler`, glue.js):** the interrupt handler already fires ~once per 20k
bytecode ops alongside the 1200-tick budget. It now ALSO, on EVERY invocation while a user cell is
in flight (`_memWatch`), reads the WASM linear-memory buffer `byteLength` and throws a typed
**`MemoryLimitError`** that unwinds the cell → `{ok:false,error:{name:"MemoryLimitError"}}`, socket
ALIVE, namespace usable.

**How the tripwire works + threshold:**
- It trips on **per-cell GROWTH**: `_memBufBase` is captured at cell entry; the trip fires when
  `buf - base > CELL_GROWTH_LIMIT_BYTES` (8 MB) OR a hard absolute backstop `buf > MAX_CELL_USED_BYTES`
  (16 MB) — whichever first. A growth-relative limit is REQUIRED because the WASM buffer is
  monotonic and never shrinks between cells; an absolute-only threshold would wedge every cell after
  the first big one and would false-trip a legitimately-large cross-cell namespace.
- Both thresholds sit BELOW `MAX_DUMP_BUFFER_BYTES` (18 MB), so the cell ABORTS before the buffer
  crosses the undumpable ceiling — the post-abort per-cell checkpoint can still dump cleanly. Live
  evidence: the v0.7 22×1MB incompressible bomb now aborts at buffer **12.1 MB** (base ~4 MB + 8 MB
  growth), eval returns `MemoryLimitError`, and the follow-up checkpoint SUCCEEDS (`store:sqlite,
  usedHeap:9.5MB`). The buffer never even reaches 18 MB.

**Why buffer byteLength, not `getMemoryUsage()`:** the first implementation probed
`getMemoryUsage().memoryUsedSize` from inside the interrupt callback. Re-entrant calls to that
heavier QuickJS memory-stats C function proved unreliable (returned 0), so the tripwire never fired
and the cell rode to QuickJS's own `out of memory`. The buffer `byteLength` is a plain ArrayBuffer
property read, always safe re-entrantly, grows monotonically with allocation, and is exactly the
quantity that approaches the 18 MB dump ceiling.

**Why probe EVERY invocation (not a coarse mask):** a tight array-push bomb crosses the limit in
only a handful of interrupt invocations; a 1-in-64 probe missed it. The interrupt fires only ~per
20k ops and the tick budget caps invocations at ≤1200/cell, so this is ≤~1200 cheap buffer reads
per cell — well under the ~1.6k/turn workerd interrupt-throttle ceiling.

**Verified (smoke 24/24 + probes):** array-push bomb `a.push(new Array(1000).fill(i))`,
string-concat bomb `s+='abcdefghij'`, object-spam bomb — ALL trip a typed `MemoryLimitError`
mid-cell, socket alive, next eval works. The v0.7 sparse-write 22MB incompressible bomb now caught
mid-cell at 12 MB (strictly better than the old post-cell `SizeAdmissionError` at 18 MB).

**Documented limitation (interrupt-cadence floor):** the tripwire can only fire on a BYTECODE
interrupt. A bomb dominated by native C builtins that yield no bytecode (e.g. a loop of
`new Uint8Array(1MB).fill(1)` — `.fill` is a C memset) emits almost no interrupts and can outrun
the probe. Such a cell either OOMs catchably (QuickJS `out of memory`, socket alive) or — with an
inner try/catch — grows the buffer large. This is the same fundamental ceiling as the loop-preempt
tick budget. Realistic alloc bombs (array/object/string accretion in a JS loop) are all
bytecode-driven and ARE caught. Not a release blocker; tracked for a future host-side buffer-grow
hook if a native-builtin bomb proves a real workload.

## Feature B — Tier-0 extensions wire-up

Wires the 5 prebuilt quickjs-wasi native extensions (`crypto`, `encoding`, `url`,
`structured-clone`, `headers`) that v0.7 shipped unused, per `docs/research/repl-env-surface.md`.

**Mechanism:**
- `scripts/prepare-extensions.mjs` copies the 5 `.so` out of `node_modules/quickjs-wasi/extensions/`
  into `src/ext/<name>.wasm` (renamed to `.wasm` because wrangler's bundler maps the CompiledWasm
  loader by file extension — a `.so` errors "no loader configured"). Bytes unchanged.
- `wrangler.jsonc` CompiledWasm rule glob extended to `src/ext/*.wasm`. `entry.mjs` imports each as a
  precompiled `WebAssembly.Module` and exposes `globalThis.__QJS_EXT_MODULES` (workerd forbids
  runtime `WebAssembly.compile` of arbitrary bytes; `loadExtension` accepts a precompiled Module).
- `glue.js` `buildExtensionDescriptors()` builds the `ExtensionDescriptor[]` in a canonical
  `EXTENSION_ORDER` and passes it to **BOTH** `QuickJS.create` AND `QuickJS.restore`.

**CRITICAL — restore re-provides the natives:** `restore()` makes a NEW WASM instance and blits the
heap back. The native extension bindings (function-table entries at fixed memory/table bases) are
NOT in the heap blit — quickjs-wasi re-instantiates each extension at the base recorded in the
snapshot BEFORE overwriting linear memory. So restore MUST pass the SAME descriptors in the SAME
order; omitting them would leave the restored heap referencing missing natives. Both paths call the
single `buildExtensionDescriptors()` with the single `EXTENSION_ORDER`.

**New VM globals (all verified live):** `crypto.subtle.digest('SHA-256', …)` (→ bytes
186,120,22,191 for 'abc'), `crypto.getRandomValues`, `crypto.randomUUID`,
`new TextEncoder().encode` / `TextDecoder().decode`, `new URL(...).hostname`, `URLSearchParams`,
`structuredClone({a:1})`, `new Headers()`.

**DETERMINISM PRESERVED (the key correctness property):** the crypto extension's `getRandomValues`
and `randomUUID` route through the WASI `random_get` import — which glue.js's WASI factory ALREADY
seeds (mulberry32). So they are deterministic and snapshot-reproducible with NO patch. We also
install a seeded JS `getRandomValues` shim on top as a belt-and-suspenders (routes through the same
seeded `__hostRandom` counter) and keep native `subtle`/`randomUUID`. `crypto.subtle` digests are a
pure function of input (no entropy) so deterministic regardless. On `clock:'real'` the shim's
`__hostRandom` returns `Math.random()`, so determinism is gated on config exactly as before.

**Verified:** seeded `crypto.getRandomValues` byte-identical across two fresh seeded sessions
(`231,199,71,87,…`), and continues deterministically across evict/restore (post-restore draw
matches the no-restore baseline). All 5 globals work AFTER evict/cold-restore (`sqlite-restore`,
`inMemory:false`), including `crypto.subtle` returning the identical digest.

## Gotchas
- Extension `.so` MUST be served as `.wasm` (wrangler loader maps by extension) and as a precompiled
  Module (workerd forbids runtime compile). url.so is ~920 KB; total upload ~3.8 MB / gz 1.25 MB.
- Mid-cell tripwire is bytecode-interrupt bound — native-builtin bombs (`.fill` on huge typed
  arrays) can still outrun it (documented limitation above; QuickJS OOM stays catchable, socket alive).
- A real cold wake (generation bump) needs the DO isolate to actually evict; in this window the
  evict-then-reconnect path (`inMemory:false`, `sqlite-restore`) already exercises the
  extension-re-instantiation restore code path, which is the load-bearing correctness check.
