# montydyn V0.1 — Final Build + Verification Report

**Deployed:** `wss://montydyn-v01.umg-bhalla88.workers.dev` (`engineHash=52d4a276…`)
**Branch:** `feat/v0.1`
**Date:** 2026-06-01
**Verification:** 4 independent live test passes over real WebSocket (errors/regression gate, config+tools, regression+latency, plus integrated smoke 24/24). montydyn-v0 left intact as baseline.

---

## TL;DR

V0.1 delivers a genuinely **dynamically-configured, stateful, error-safe** JS environment. The two highest-value bugs — **BUG-1 (error wedge)** and the V0 cold-restore correctness — are fixed and regression-clean. Dynamic config, host tools, console capture, seeded clock/RNG determinism, and structured value previews all work live.

But **two of the six claimed bug fixes do not hold on the deployed build**:
- **BUG-2/BUG-4 (pre-dump GC / memory reclaim) — NOT FIXED.** Freed memory is never reclaimed; the live image only grows, and a size-guard trip permanently wedges checkpointing. Confirmed independently by two testers.
- **BUG-3 (CPU/wall budget) — PARTIAL.** A typed `TimeoutError` fires for some loops but **not reliably** for real CPU-bound or truly-empty loops — those still ride to the DO wall limit and close with WS **1006**.

**Recommendation: MERGE `feat/v0.1` → `main`** (it's a strict, large improvement over V0 with no regressions), but **do not advertise BUG-2/BUG-4 as fixed**, and **gate the V1 facets work on landing a real GC/compaction fix and a reliable interrupt budget.**

## 1. V0.1 Status — claim-by-claim

### Is BUG-1 truly fixed? — YES. Unambiguously. ✅
The primary mergeability gate is **GREEN (errors suite 17/17, smoke 24/24)**.
- Every error class — `throw`, `SyntaxError`, `ReferenceError`, `TypeError`, throw-in-expression, stack-overflow `RangeError` — returns structured `{ok:false, error:{name,message,stack}}`.
- The socket **stays open** every time (no WS close); the `eval_critical` mutex is **always released**.
- The **next eval works** after any error (e.g. `2+3`→5 immediately after a throw; `7*6`→42 after a syntax error).
- `{t:reset}` recovers a wedged/errored session (`epoch:1`, namespace cleared, `typeof x`→"undefined"); fresh reconnect on same id also recovers.
- **Concurrency clean:** 20 concurrent evals with 5 interleaved throws — all 20 replied (no hang), 5 throwers `ok:false`, counter on the 15 non-throwers exactly 15.

Root-cause fix is real and verified: the wasm-bindgen `evalCode` boundary now returns a JSON **string** (never throws across the wasm boundary), so the mutex can't be orphaned.

### All 6 bugs resolved? — NO. 4 of 6.

| Bug | Claim | Verified status |
|---|---|---|
| **BUG-1** eval-mutex deadlock on uncaught error | Fixed | ✅ **FIXED** (17/17, concurrency clean) |
| **BUG-2** un-checkpointable after size-trip | Fixed (GC before dump) | ❌ **NOT FIXED** — size-trip permanently wedges checkpointing |
| **BUG-3** no CPU/wall budget → WS 1006 | Fixed (tick + wall budget) | ⚠️ **PARTIAL** — fires for some loops; real CPU-bound & empty loops still 1006 |
| **BUG-4** live image never reclaimed | Fixed (runGC before dump) | ❌ **NOT FIXED** — high-water mark only grows; freed mem never shrinks image |
| **BUG-5** `performance.now()` unseeded | Fixed (seeded off same clock) | ✅ **FIXED** — control vs restored byte-identical (`MATCH true`) |
| **BUG-6** restoreSource mislabeled R2 | Fixed (`r2-restore` vs `sqlite-restore`) | ✅ **FIXED** — R2-backed restore now reports `r2-restore` |

**BUG-2/BUG-4 evidence (the hard failure):**
- Allocate 30MB then `delete`: checkpoint `sizeRaw` 1,310,748 → 32,768,028 and **stays pinned at 32,768,028** across the delete and 4+ subsequent evals. Same with a 22MB array (22,151,196, never shrinks).
- `new Uint8Array(60MB)` → `SizeAdmissionError: live image 64225280 > 52428800`; then `delete` → still `SizeAdmissionError`; then `40+2` → **still** `SizeAdmissionError`. Session permanently un-checkpointable — **identical to original BUG-2**.
- Regression tester confirms independently: after `delete globalThis.big` + 8 cells, gzip image flat at ~3,778,700, `store` never returns from `r2` to `sqlite`. `gc()` not exposed in the VM. The runGC-before-dump change is **not in effect on the deployed build**.

**BUG-3 evidence (the partial):**
- ✅ `while(true){}` returned a typed `TimeoutError` in ~1.4s, socket open (`readyState=1`), state survived, in the smoke/errors-gate pass.
- ❌ But config+tools tester saw empty `while(true){}` → **WS 1006** (DO killed), and a CPU-bound non-allocating `for(i<1e12){x=(x+i)%7}` (with `cellBudgetTicks:30000`) **ran past 40s, no TimeoutError**, riding to the DO wall limit.
- The build manifest's own open issue concedes the interrupt handler is sparse for zero-body loops. Net: budget config is accepted/persisted and works for value-touching loops in the calibrated path, but it is **not a reliable safeguard** across loop shapes. Treat as best-effort, not a guarantee.

## 2. Did anything regress vs V0? — NO. ✅

- **State survival:** eval→evict→restore intact; `x=42`, `inc()=43`; `restoreSource="sqlite-restore"`, `inMemory:false`.
- **50-cell soak with 5 mid-run evicts:** byte-perfect (`sum=1275`, `log.length=50`, first/last `1`/`50`), all restores `sqlite-restore`, no drift.
- **Latency holds** (sub-second p50 on all hot paths):

| Path | p50 | p90/tail | V0 baseline | Verdict |
|---|---|---|---|---|
| Warm-evict sqlite-restore | 172–236ms | 543–860ms | ~165–200ms | PASS |
| Soak restore (in 50-cell run) | 158–162ms | — | 162–174ms | PASS (flat) |
| Real cold-wake (75s idle, gen 1→2) | 179ms | — | ~165ms | PASS |
| R2-overflow restore (3.5MB) | 2.1–4.5s | — | 0.5–0.9s | functional, slower this run |

- **BUG-6 fixed without regressing** the R2 swap-then-delete (fresh key per cell, durable across evict).
- Tail spikes are WS edge round-trip jitter, consistent with V0. No DO crash / WS 1006 on any *valid* input. The only 1006s observed were from the BUG-3 runaway-loop gap, not from regressions.

R2-overflow restore p50 (2–4.5s) was slower than V0's sub-second this run; not a hard regression (small N, single run, network-bound) but worth a re-measure.

## 3. Remaining bugs / gaps — ranked

1. **[P0 — blocker if in scope] BUG-2/BUG-4: no memory reclaim before dump.** Live image only grows; size-trip permanently wedges checkpointing; R2-overflowed sessions never return to the SQLite hot path. The runGC change is not on the deployed build. *Impact:* any session that briefly spikes memory is cost-pinned to R2 forever or becomes un-checkpointable — a real durability/cost liability.
2. **[P1] BUG-3: budget doesn't reliably preempt.** Real CPU-bound and truly-empty loops still WS-1006 the DO. *Impact:* a single bad cell can kill the DO; the "typed TimeoutError instead of 1006" promise is not dependable.
3. **[P2] Host-tool *state* not persisted across restore.** `host.kv.put` data is lost on cold wake (`kv.get`→`null`, `keys`→`[]`); only the tool *binding* re-registers. kv is the headline "persisted" demo tool, so this is misleading.
4. **[P3] Fetch allowlist accepted but inert.** `fetch:[...]` is persisted but never installs a `fetch` global — fail-safe, but the config surface implies a capability that doesn't exist yet.
5. **[P3 — cosmetic] Error-*value* preview drops message/stack.** An error returned as a value surfaces only `{name}` (thrown errors carry full name/message/stack).
6. **[P3 — observability] R2-overflow restore latency** elevated this run (2–4.5s vs V0 sub-second); confirm not a real regression.

## 4. Is V0.1 a usable & useful dynamically-configured stateful JS environment? — YES, with caveats. ✅

It clears the bar for "the foundation V1 facets build on":
- You can **create a session with declarative config** (clock, RNG seed, capture, host tools), run stateful multi-cell code, and that config + state **survives cold restore** at sub-second p50.
- It is now **error-safe**: no throw, syntax error, or stack overflow can wedge the kernel — the single biggest blocker to a usable REPL/agent-execution surface is gone.
- It is **deterministic** where it claims to be (seeded clock + RNG byte-identical across restore) — exactly what reproducible/replayable agent execution needs.
- A **reconnect-safe client SDK** (`sdk/index.mjs`) wraps it cleanly (eval/reset/gen/create/evict/close, re-sends config on reconnect).

The caveats that keep it from "production-hardened": memory is never reclaimed (BUG-2/4), runaway code can still kill the DO (BUG-3), and host-tool data doesn't persist. None of these break the *happy path*, but all three are reachable by ordinary or adversarial workloads.

**Verdict:** usable and useful as a V1 *foundation*; not yet safe to expose to untrusted/adversarial code or memory-spiky workloads.

## 5. Merge recommendation + next steps

### Recommendation: **MERGE `feat/v0.1` → `main`.** ✅
Rationale: V0.1 is a strict superset of V0 with **zero regressions**, fixes the #1 blocker (BUG-1) cleanly, and adds the entire dynamic-config/host-tool/determinism surface verified live. Holding the merge for BUG-2/BUG-4 would block a large amount of good, independent work.

**Merge with eyes open — required before/at merge:**
1. **Correct the changelog/README claims:** mark BUG-2/BUG-4 as *not landed on deploy* (the source change appears to not be in the built wasm) and BUG-3 as *best-effort, not guaranteed*. Do not ship docs that say these are fixed.
2. Add a known-issues note for host-tool state non-persistence and the inert fetch allowlist.

### Next steps (priority order)
1. **[P0] Land a real memory-reclaim path** before dump: confirm `kernel.runGC()` is actually compiled into the deployed wasm and that the linear-memory high-water mark can shrink (or that the dump serializes only live objects, not the arena). Add a regression test: allocate→delete→checkpoint `sizeRaw` must return toward baseline; size-trip must be recoverable after `delete`. This is the single most important fix for cost + durability.
2. **[P1] Make the interrupt budget reliable** across loop shapes (empty `while(true){}` and CPU-bound non-allocating loops). Verify the tick budget trips well under the ~30s DO wall limit for *all* loops, and that a 1006 is never the outcome of a single bad cell.
3. **[P2] Persist host-tool state** (at least kv) in snapshot meta, or document loudly that host tools are stateless across restore.
4. **[P3] Either wire the fetch allowlist** to a real (gated) egress path or remove it from the accepted config surface until implemented.
5. Re-measure R2-overflow restore latency to confirm point 6 above is jitter.

### Proceed to V1 facets?
**Conditional YES.** The foundation is sound enough to start V1 facet design in parallel. But **gate any V1 facet that relies on memory bounds, long-running/untrusted execution, or persistent host-tool state on fixes #1–#3 above.** For facets that only use the happy path (config + stateful eval + determinism + error-safety), V0.1 is ready to build on today.
