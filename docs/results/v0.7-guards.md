# v0.7 — Stdlib Guards (the three OOM-cliff guards on top of v0.6)

> Worker `montydyn-v07` (https://montydyn-v07.umg-bhalla88.workers.dev).
> Branch `feat/v0.7`. AE dataset `montydyn_kernel`, R2 prefix `v07/`. 2026-06-01.
> Built on v0.6 (`v0.6/` copied verbatim to `v0.7/`, then the 3 guards applied).

## What v0.7 adds

v0.6 proved configurable in-VM stdlib (esbuilt libs -> Text module -> evaled into the
QuickJS heap at create -> snapshot-persisted) is **GO, viable-with-limits**. Its one
dangerous breaking point: the **snapshot OOM cliff at ~24-30 MB RAW incompressible heap**
— which is BELOW the old `MAX_DUMP_BUFFER_BYTES` ~45 MB ceiling — so a heavy injected
stdlib (mathjs, a big synthetic bundle) or a large in-VM allocation got **admitted** to
dump and then OOMed the isolate (silent WS 1006 / DO kill) during the ~2-3x serialize
transient. v0.7 ships the three guards that close that gap and make the death a clean,
typed, socket-alive rejection. Nothing else about v0.6 changes.

All other v0.6 behavior is preserved: stdlib inject+persist, determinism (seeded
clock/RNG/crypto), kv, fetch+allowlist, loop preempt (1200-tick), deep restore,
AE telemetry.

## The three guards

### GUARD 1 — config cap on total injected stdlib source (`src/glue.js`)

- `MAX_STDLIB_SOURCE_BYTES = 500 * 1024` (the verified-safe ~500 KB envelope).
  At create, `enforceStdlibSourceCap(modules)` sums the iife source length of every
  resolved module and throws a typed `SizeAdmissionError` (socket alive) **before any
  eval-into-heap** if the total exceeds the cap. A heavy bundle can therefore never grow
  the buffer toward the OOM cliff in the first place.
- `MAX_INLINE_STDLIB_BYTES = 2 * 1024 * 1024` fences the separate, lower inline-blob
  cliff: a single `config.stdlib` source string >= 2 MB is rejected with a typed
  `SizeAdmissionError` (the multi-MB one-shot compile transient WS-1006s around ~2.1 MB).
  The per-module `config.modules` path avoids that cliff entirely. (v0.7 also wires the
  inline `config.stdlib` path: a string blob is evaled AFTER the bundle modules, with its
  own try/catch so a throw in the blob never bricks the kernel.)

### GUARD 2 — mathjs (and any >29x-amplification lib) is OPT-IN only

- `scripts/build-stdlib.mjs`: new `OPT_IN = ["mathjs"]`. Opt-in libs are bundled (so they
  CAN be named explicitly) but are written to `stdlib-meta.js` under `optIn`, and the
  default set (`meta.modules`) excludes them.
- `glue.js` `resolveModules`: `config.modules: true` returns the default set MINUS the
  opt-in set, so mathjs **never** loads from `true`. It loads ONLY when explicitly named
  in `config.modules: ["mathjs"]` — and even then it still counts against GUARD 1's source
  cap (mathjs alone is ~617 KB iife > 500 KB, so it trips the cap on its own).
- `mathjs@12.4.3` added to devDependencies + `stdlib-src/mathjs.js` (`globalThis.math`).

### GUARD 3 — lower `MAX_DUMP_BUFFER_BYTES` 45 -> 18 MB (THE KEY FIX) (`src/glue.js`)

- `MAX_DUMP_BUFFER_BYTES = 18 * 1024 * 1024`. The dump side refuses ANY snapshot once the
  monotonic WASM linear-memory buffer exceeds 18 MB, returning a typed `SizeAdmissionError`
  (socket alive, prior committed snapshot intact) BEFORE the ~2-3x snapshot+serialize
  transient can run. 18 MB sits comfortably BELOW the real ~24-30 MB OOM cliff, so any
  buffer approaching the cliff is clean-rejected instead of crossing it.
- `MAX_RESTORE_RAW_BYTES` lowered 45 -> 18 MB in lockstep (kept pinned to the dump ceiling:
  "if it could be dumped, it can be restored" — and nothing bigger than 18 MB can ever be
  written, so no legitimate snapshot is stranded).
- `SCRUB_MAX_BUFFER_BYTES` lowered 32 -> 16 MB so the arena scrub stays below the dump
  ceiling (a scrub must never run on a buffer the dump side would refuse).

The used-heap guard (`MAX_USED_BYTES` 50 MB) and the spike-then-free un-wedge from v0.2 are
unchanged: a session that spikes its *buffer* past 18 MB then frees can still NOT durably
checkpoint (WASM memory is monotonic), but it keeps working in-memory and reset recovers —
exactly the documented, safe behavior, now with a much lower (and OOM-safe) ceiling.

## Live smoke — montydyn-v07 (`v0.7/smoke-v07-guards.mjs`, 13/13 PASS)

### (a) default stdlib loads + works + survives REAL evict (regression)
`create {modules:true}` -> `loaded=[lodash,dayjs,nanoid,uuid,zod]`; `_.chunk([1..5],2)=[[1,2],[3,4],[5]]`,
`dayjs=2020-06-15`, `zod.safeParse=true`. Set `userState=600`, evict + 75 s real idle ->
reconnect: `generation 1->2`, `restoreSource=sqlite-restore`, `_.chunk([9,8,7,6],2)=[[9,8],[7,6]]`,
`userState=600` survived, reconnect `create` shows `stdlib.loaded=[]` (libs rode the snapshot,
NO re-inject). **PASS.**

### (b) >500 KB selected source -> typed cap error, socket alive
`create {modules:["lodash","zod","mathjs"]}` (788,893 bytes) ->
`{ok:false, error:"SizeAdmissionError: selected stdlib source 788893 bytes > MAX_STDLIB_SOURCE_BYTES 512000 ..."}`.
Follow-up `{t:gen}` returns `ok:true` (**socket alive**). A safe `create {modules:["lodash","dayjs"]}`
on the same session then loads fine (recovers). **PASS.**

### (c) mathjs only loads when explicitly opted in
`create {modules:true}` -> `loaded` excludes mathjs, `optIn=["mathjs"]`, `typeof math === "undefined"`.
Naming mathjs explicitly (`modules:["mathjs"]`, 631,999 bytes) is rejected by GUARD 1
(`SizeAdmissionError ... > MAX_STDLIB_SOURCE_BYTES 512000`), socket alive. **PASS.**

### (d) THE KEY FIX — heap toward the old ~24-30 MB OOM cliff -> typed reject, socket ALIVE
A cell allocated ~22 MB of incompressible bytes retained on a global, growing the WASM
linear buffer to **24,379,392 bytes (~24.4 MB)** — squarely inside the old 24-30 MB silent-OOM
band that lived BELOW the previous 45 MB ceiling. The per-cell checkpoint **clean-rejected**:

```
checkpoint.error = "SizeAdmissionError: linear-memory buffer 24379392 > MAX_DUMP_BUFFER_BYTES 18874368;
                    refusing snapshot (WASM memory cannot shrink in place and dumping/scanning the full
                    buffer would risk an uncatchable OOM). reset to recover."
```

The socket stayed **ALIVE** (no WS 1006: the alloc-eval returned, a follow-up `{t:gen}`
returned `ok:true`), and the VM remained usable in-memory (`1+1 === 2`). Pre-v0.7 this same
24.4 MB buffer would have been admitted to dump (it is < 45 MB) and OOMed the isolate during
the serialize transient. **oomCliffNowCleanRejects = true. PASS.**

## Verdict

The three guards are live and verified. The single dangerous v0.6 breaking point — the
silent ~24-30 MB OOM cliff below the dump ceiling — is **closed**: a >ceiling incompressible
heap now returns a typed `SizeAdmissionError` with the socket alive instead of a silent WS
1006 / DO kill. Heavy / opt-in stdlib selections are rejected at create with typed errors,
socket alive, and recover via a fresh safe create. All v0.6 behavior preserved.
