## Architecture (as built)

```
Worker entry  (src/supervisor.js default export)
  └─ shardFor(sessionId) -> 1 of 64 SupervisorDO shards   ← scale guardrail
       SupervisorDO (normal SQLite DO, new_sqlite_classes)
         ├─ worker_loaders binding LOADER (kernel class, codeId-versioned)
         ├─ ROUTING: facetName = "k:<tenant>:<sessionId>" -> ctx.facets.get(...)
         ├─ ALARMS/TTL: idle facets hard-evicted via ctx.facets.abort (facets can't alarm)
         ├─ WS HIBERNATION: supervisor accepts the client socket + tags (t:/s:),
         │   webSocketMessage proxies each frame to the facet via RPC (handleMessage)
         ├─ per-supervisor facet CAP (128) — bounded, never thousands under one
         └─ control-plane SQLite: sessions registry (isolated from facets)
            └─ KernelFacet (dynamically loaded DO, src/facet-kernel.js)
                 ├─ quickjs.wasm via {wasm} Worker-Loader module (CompiledWasm)
                 ├─ ported v0.8 GlueKernel (src/glue.js, verbatim) — wasm import rewritten
                 │   to ./qjs/index.js at bake; engine handed in via globalThis.__QJS_MODULE
                 ├─ own ctx.storage.sql: snap (heap gz + manifest) + meta (gen/config)
                 └─ per-cell synchronous checkpoint -> cold-restore across abort (no replay)
```

### Connection model (from the WS-hibernation spike)
The SUPERVISOR holds the hibernatable client WebSocket (acceptWebSocket works on a normal
DO) and proxies each frame into the facet via RPC. A facet-held client socket does NOT work
(DataCloneError over RPC; no inbound frame delivery via facet.fetch relay). Proven live:
`servedBy:"supervisor-proxy->facet"`.

### codeId / engine-hash versioning (the loader cache foot-gun)
`scripts/bake-modules.mjs` computes `sha256(glue + facet + quickjs.wasm + dist)` and sets
`KERNEL_CODE_ID = "kernel-<hash16>"` (the `LOADER.get` cache key) and `ENGINE_HASH =
"v1-<hash16>"` (glue's restore guard). Any source change yields a new codeId -> fresh
compile (no stale-isolate reuse) AND a new engine hash -> clean `EngineHashMismatchError`
instead of a corrupt restore. Live codeId: `kernel-88f1a9c3b961fd94`.

## Ported (V1.0 scope)
- Real v0.8 GlueKernel (quickjs-wasi 3.0.0 + real ~1.6MB quickjs.wasm) running **inside the facet**.
- SQLite-first heap snapshot/restore **in the facet's own SQLite**; atomic per-cell checkpoint (single UPSERT, monotonic `gen`).
- Cold-restore across `ctx.facets.abort` (no replay), with entropy counters + host-kv + config carried in the manifest.
- Core safety: loop budget (1200 ticks -> typed TimeoutError), mid-cell used-heap tripwire, size-admission guards incl. the 18MB dump ceiling (`MAX_DUMP_BUFFER_BYTES`), engine-hash restore guard.
- Seeded determinism (clock + mulberry32 RNG), reproducible across restore.
- Multi-tenant isolation (facet-per-`tenant:session`, isolated SQLite each).
- Supervisor routing + sharding (FNV-1a over sessionId -> 64 shards, 128-facet cap each) + alarm-driven idle TTL eviction (5min) — alarms on the supervisor since facets can't set them.
- WS-hibernation proxy model (supervisor holds socket, RPC per frame to facet).
- ASYNC eval + fetch pump carried over from P3 (host.fetch defaults disabled; `globalOutbound:null` sandboxes the facet).

## Deferred to V1.1 (noted, NOT built)
- **Configurable in-VM stdlib** (lodash/dayjs/zod/nanoid/etc.): glue reads `globalThis.__STDLIB_BUNDLE`/`__STDLIB_META`; intentionally unset in the facet, so `stdlibBundle()` -> `{}`. Additive, already proven in v0.8.
- **Tier-0 native extensions** (crypto/encoding/url/structured-clone/headers): glue reads `globalThis.__QJS_EXT_MODULES`; unset, so `buildExtensionDescriptors()` -> `[]`. Core determinism is unaffected (seeded clock/RNG live in the WASI shim, not the extensions); a JS seeded `crypto.getRandomValues` shim still installs via REBIND_SRC.
- **Adaptive keep-warm** (predictive facet pre-warm). Current model: per-cell synchronous checkpoint + supervisor idle-TTL eviction.
- **Mediated fetch gateway**: pass a service binding as the loader's `globalOutbound` so host.fetch can be allow-listed per tenant (today egress is blocked at the facet).
- **R2 large-object offload**: `SNAPSHOTS` binding is wired (namespace `v1/` keys) but snapshots currently live SQLite-first; >18MB is refused rather than spilled.

## Gotchas / notes for the next builder
- glue.js imports `from "quickjs-wasi"`; the bake rewrites that single specifier to `./qjs/index.js` (the dist ships as `{js}` under `qjs/`). glue is also shipped as a `glue.js` loader module (facet-kernel imports `./glue.js`).
- The `{wasm}` module type is the ONLY wasm-delivery path inside a facet (raw-bytes compile blocked). Engine handed to glue via `globalThis.__QJS_MODULE` + `globalThis.__ENGINE_HASH` set at the top of facet-kernel.js before any kernel op.
- Cold-start race: first `/health` after deploy returned a transient `error code: 1042`; retried clean. WS first-frame on a cold supervisor isolate may need one retry (matches the spike note) — the smoke retries once.
- The size-bomb sometimes surfaces as QuickJS's own `InternalError: out of memory` (a catchable, contained OOM) rather than the typed `MemoryLimitError` tripwire, depending on whether the linear-memory BUFFER (vs the malloc arena) crossed the 8MB/16MB thresholds at an interrupt sample. Either way the cell is contained, socket alive, kernel usable — the safety invariant holds; only the label differs.
- `src/quickjs.wasm` is force-added to git (global gitignore excludes `*.wasm`).
- `src/modules.gen.js` is gitignored (regenerate with `npm run bake`).

## (2) Isolation + failure-isolation: STRONG PASS

- **Data/heap isolation:** two layers, both verified. (a) Per-tenant facet-name keying -> isolated `ctx.storage.sql` per session; (b) the QuickJS WASM VM boundary — sandboxed user code has **no** `env`/`SUPERVISOR`/`LOADER`/`ctx`/`process`/`fetch`/`facets`/`storage`/`require` (all undefined; `globalThis` secret-scan returned `[]`). Egress blocked (`globalOutbound:null`).
- **Failure isolation (decisive):** **10 concurrent loop+alloc bombs across multiple shards** ran while a victim session returned its correct state (`safe=777`), HTTP 200, warm, uncorrupted, in 585ms. A loop bomb on `co/sess4` did not affect siblings `sess55`/`sess99` **on the same shard** (both returned correct state in ~500–600ms). After an OOM bomb, sibling `t1/s2` cold-restored cleanly (`x=2`) from its own SQLite. **No misbehaving facet ever took down the supervisor or affected a sibling's state/availability.**

## (3) Scale: GOOD — sharding works, no single-supervisor bottleneck

- **Sharding:** FNV-1a(sessionId) -> 64 SupervisorDO shards, cap 128 facets/shard. 50 test sessions spread across 46 distinct shards (max 2 facets/shard) — load distributes as designed; the ~1k-req/s single-supervisor guardrail was never approached.
- **Throughput (single region):** 50 sessions/40 concurrent warm = **36 req/s, 50/50 200, 0 errors**; 60 sessions cold = 0 failures; 200 req/60 sessions/50 concurrent = 55 req/s, ~1% transient 5xx; 300 req/60 concurrent repeat = 0 non-200. Cold-create with 30 concurrent = 23 req/s, 49/50 200.
- **Error rate:** ~1% non-deterministic transient 500s on cold/overload, no captured body, not reproducible on retry — aligns with the documented cold-start RPC-hiccup risk (v1-direction §4.5), **not** a kernel/routing fault. Supervisor `/health` stayed green throughout.

## (4) Ported vs deferred

**Ported & verified (V1.0):** real v0.8 GlueKernel (quickjs-wasi 3.0.0 + ~1.6MB `quickjs.wasm` via `{wasm}` CompiledWasm), SQLite-first heap snapshot/restore in the facet's own SQLite, atomic per-cell checkpoint (single UPSERT, monotonic gen), cold-restore across `ctx.facets.abort` with no replay, loop budget (1200 ticks -> typed `TimeoutError`), mid-cell used-heap tripwire, size-admission guards incl. 18MB dump ceiling (`MAX_DUMP_BUFFER_BYTES` — confirmed firing: buffer 267583488 > 18874368 -> refused, prior snapshot intact), engine-hash restore guard, seeded determinism (clock + mulberry32, byte-identical across restore), multi-tenant facet-per-session isolation, supervisor routing + FNV-1a sharding, alarm-driven idle TTL, WS-hibernation proxy model, async eval + fetch pump (P3), `globalOutbound:null` sandbox, codeId/engine-hash content versioning.

**Deferred to V1.1:** configurable in-VM stdlib (`__STDLIB_BUNDLE`/`__STDLIB_META` unset -> `{}`); Tier-0 native extensions crypto/encoding/url/structured-clone/headers (`__QJS_EXT_MODULES` unset -> `[]`; seeded JS crypto shim still installs); adaptive keep-warm / predictive pre-warm; mediated fetch gateway (per-tenant allow-listed `host.fetch` via a service binding as loader `globalOutbound`; egress currently fully blocked); R2 large-object snapshot offload (`SNAPSHOTS` binding wired with `v1/` keys but snapshots are SQLite-first; >18MB refused, not spilled).

## (5) Bugs / risks (ranked)

1. **[MED] ~1% transient cold-start 5xx under concurrent load.** Non-deterministic, no error body, clears on retry. Likely cold-start/overload RPC hiccup. **Mitigation:** first-call retry/warmup on the supervisor entry; track in V1.1.
2. **[LOW] WS first-frame timeout on a cold supervisor isolate.** Cold-start race; retries succeed. **Mitigation:** generous first-frame timeout or warmup ping (smoke already retries once).
3. **[LOW] Size-bomb label non-determinism.** Sometimes surfaces as QuickJS catchable `InternalError: out of memory` instead of the typed `MemoryLimitError`, depending on whether linear-memory buffer vs malloc arena crossed thresholds at the interrupt sample. **Safety invariant still holds** (contained, socket alive, kernel usable) — only the error label differs.
4. **[LOW / operational] codeId loader-cache foot-gun.** The loader caches the compiled isolate by codeId, NOT by config-closure contents (proven in spike: same codeId + changed source -> silent stale isolate). **Already mitigated** by content-hash codeId (`sha256(glue+facet+wasm+dist)`); the discipline (bump codeId on any source change) must be maintained.
5. **[INFO] Egress fully blocked at the facet.** Intentional for V1.0; the mediated fetch gateway is the V1.1 unblock path.
6. **[INFO] Per-supervisor 128-facet cap + 64 shards = 8192 concurrent-facet soft ceiling per region.** Adequate for V1.0; revisit shard count if approached.
