## Goal

Make the kernel **observable from the server side** (not just client RTT), then run a
**deep-test pass** — durability under fault injection, scale/concurrency limits,
adversarial containment, idle-eviction soak — and use the server-side data to answer the
open v0.4 question: **is the cold-start tail platform-bound or kernel-fixable?**

## 1. Observability — LIVE

Two server-side sinks, both confirmed working against the live worker:

### Analytics Engine (AE)
- Binding: `analytics_engine_datasets` `AE` -> dataset `montydyn_kernel` (`v05/wrangler.jsonc`).
- One datapoint per op, emitted from `KernelDO::emit()` (`v05/src/lib.rs`). The worker crate
  has **no first-class AE binding**, so `writeDataPoint` is called via `js_sys` Reflect on the
  Env JsValue; `emit` is a no-op if the binding is absent (safe degrade).
- **Schema** (queried back live and confirmed):
  - `index1` = DO id (used for HyperLogLog fleet-size estimates).
  - blobs `[op, restoreSource, store, errorName, configClock, valueType, label]` -> `blob1..blob7`.
  - doubles `[totalServerMs, readMs, sizeRaw, sizeGz, usedHeap, cell, generation, gunzipMs,
    instantiateMs, growCount, nChunks, ok]` -> `double1..double12`.
- **Ops instrumented:** `create, restore, eval, error, timeout, fetch, fetch-blocked,
  size-reject/checkpoint, gen, ping, evict, reset`. `ok=1` on success, `0` on error/timeout.
  `restore` carries `restoreSource` (cold/lazy `sqlite-restore` vs R2). `restoreSource` =
  `sqlite-restore` on the SQLite hot path.

### Workers Logs
- `observability.enabled` kept on. Every op also `console.log`s one structured JSON line
  `{ev:op, ...}` with the full field set, so Logs is independently queryable.

### How to query (AE SQL API)
`POST https://api.cloudflare.com/client/v4/accounts/{accountId}/analytics_engine/sql`
(token sourced via `set -a; . .env; set +a` — never printed). Examples:

```sql
-- op taxonomy + success rate
SELECT blob1 AS op, count() AS n, avg(double12) AS ok_rate
FROM montydyn_kernel GROUP BY op ORDER BY n DESC;

-- error/timeout histogram by typed name
SELECT blob1 AS op, blob4 AS errorName, count() AS n
FROM montydyn_kernel WHERE double12 = 0 GROUP BY op, errorName;

-- cold-restore phase distribution (note: server timings read 0, see caveat)
SELECT blob2 AS source, count() AS n,
       quantileWeighted(0.95)(double1, 1) AS p95_serverMs,
       avg(double10) AS avg_growCount, max(double10) AS max_growCount
FROM montydyn_kernel WHERE blob1 = 'restore' GROUP BY source;

-- fleet scale (distinct DOs)
SELECT uniq(index1) AS approx_sessions FROM montydyn_kernel;
```

Notes that bit us, now documented:
- **Ingestion lag ~30-90s.** An immediate query returns `rows:0` (schema present, data empty);
  re-query after a minute.
- **AE samples under load** — use `count()` (auto-reweighted), not raw row count, at high volume;
  exact at smoke volume.
- **Percentiles:** `quantileWeighted(p)(col, 1)` works; bare `quantile()` is rejected.
- **Timestamps** are second-granularity.

## 2. DEEP-TEST verdict

Smoke regression: **`node smoke.mjs` -> 62/62 PASS.** A live v05 session was driven through
every op type (create, ping, gen, eval, error, timeout, fetch, fetch-blocked, evict,
cold-restore eval, reset); state survived eviction (a value restored on cold eval at cell 6),
and all 10+ distinct ops read back from AE.

### 2a. Durability under fault injection — NO HOLES (11/11 PASS, suite `v05/chaos.mjs`)
- **F1 crash/evict at every phase (5/5):** forced cold restore via `{t:evict}` + fresh-id
  reconnect; asserted on server `generation`/`committedCell`. Restored to last COMMITTED cell
  in every phase — after-eval (n=1), mid-multi-checkpoint (lands exactly at last committed,
  n=4), right-after-checkpoint double cold-wake (idempotent, n=5, no drift). No half-state.
- **F2 rapid evict+eval interleave (PASS):** 12 iterations evict->fresh-socket->mutate; 12/12
  cold `sqlite-restore`; final acc=12 exactly, strictly monotonic. No lost/dup mutation.
- **F3 engine-hash mismatch (PASS, genuinely triggered):** snapshot under correct engine,
  then redeployed with mangled `__ENGINE_HASH`. Restore **rejected cleanly** with typed
  `EngineHashMismatchError`, **socket stayed alive (no WS-1006)**, no corruption; redeploying
  the correct engine restored losslessly (m=555) — proving the rejected mismatch never
  destroyed committed state.
- **F4 reset recovers wedged state (3/3):** grew WASM buffer ~52MB past the dump ceiling ->
  checkpoint rejected with typed `SizeAdmissionError`, socket alive, `gen` still responds.
  `reset` cleared the namespace, bumped epoch, checkpointing worked again.

**Durability conclusion:** commit-ordering (`committedCell` advances only inside the atomic
write-coalesced turn) + engine-hash + size-admission guards all held under adversarial load.
No torn/missing/lost snapshot. One residual is **by-design** (v0.2): a buffer grown past ~45MB
can no longer durably checkpoint until `reset`, but stays alive in-memory and resets cleanly.

### 2b. Scale / concurrency — STABLE, MUTEX PROVABLY CORRECT (`v05/bench/scale.mjs`)

| scenario | config | throughput | error rate | DO kills | client RTT p50/p90/p99/max (ms) |
|---|---|---|---|---|---|
| manysessions (60 DOs) | 60 sess x 4 eval + evict + restore = 300 ops | 77 ops/s | 0.00% | 0 | 171 / 216 / 251 / 1741 |
| onesession (race) | 200 racing evals on ONE DO | 35 eval/s | 0.00% | 0 | — |
| sustained (open-loop) | 25 sess, 40s, 60 ops/s target | 59.8 ops/s | 0.00% | 0 | 168 / 232 / 315 / 1061 |

- **onesession correctness:** EXACT final counter = 200; cells strictly monotonic 0..199 (no
  gaps/dupes); 200 unique counter values (true serialization); FIFO replies; socket alive.
  Confirmed across 120/150/200-racer runs — all EXACT.
- **No DO kills, no WS-1006, 0% error** at up to 200 racing evals on one DO, 80 concurrent
  sessions, 40s sustained ~60 ops/s. Latency stable (p50 ~170ms, p99 ~250-315ms = WS RTT + DO
  wake), matching v0.4's network-bound floor. Cold-restore round-trips in manysessions added no
  error or kill.
- Server cross-check: AE recorded the full op taxonomy with ~500 distinct DO ids (HLL),
  confirming fleet scale.

### 2c. Adversarial containment — NO ESCAPE; two contained-but-imperfect recovery gaps (`v05/fuzz.mjs`)
**Cleanly contained** (socket alive + kernel usable next cell): stack bombs (RangeError),
most memory bombs (InternalError / typed SizeAdmissionError), CPU bombs in isolation
(TimeoutError), gigantic-JSON, ReDoS x2, weird-unicode/huge-source, throw-in-async, unhandled
rejection, throw primitive/null, syntax error, global clobbering, host-internals probe,
WebAssembly codegen blocked, Function-constructor (same global, no escape), binary frame +
garbage JSON. **Determinism: 8/8 random seeded programs byte-identical across evict + cold
restore.** Mutex: 100 racing evals -> exact counter, monotonic, no dupes. Fleet: 40 sessions,
160 ops, 0 errors, 0 kills (p50 176ms / p99 320ms).

**No real sandbox escape:** `process`/`require`/`Deno` undefined, runtime `WebAssembly.compile`
blocked, `Function('return this')` returns the VM global, not the host.

### 2d. Idle-eviction soak + server-side latency (`v05/bench/idle-soak.mjs`, `ae-latency.mjs`)
- **Real idle-eviction soak — 4/4 PASS.** Same DO id, set state -> close WS -> idle 72s (genuine
  hibernation) -> reconnect -> verify. Generation bumped every cycle (1->2->3->4->5), every wake
  a real `sqlite-restore`, restored state exact each time. Reconnect RTT 168-228ms.
- **Long multi-cell session — INTEGRITY INTACT.** 120 cells, 6 evictions (3 sim + 3 real idle),
  0 integrity failures; final acc=7260 = Σ(1..120) exactly, n=120, histLen=120.

## 3. Cold-start: IS THE TAIL PLATFORM-BOUND?

**Yes — server-side AE data confirms the cold tail is platform/network, not our restore.**

Server `totalServerMs` and `readMs` read **0 at every percentile for every op** (fresh,
sqlite-restore, warm) — including p99/max across ~240 datapoints, AND including the genuine
high-generation idle-eviction restores (gen 2/3/4/5: maxServerMs=0, maxReadMs=0).

| op / source | n | server p50 | p90 | p99 | max | growCount avg/max |
|---|---|---|---|---|---|---|
| eval / sqlite-restore (cold) | 58 | 0 | 0 | 0 | 0 | 1 / 1 |
| eval / warm | 61 | 0 | 0 | 0 | 0 | 0 / 0 |
| eval / fresh | 121 | 0 | 0 | 0 | 0 | 0 / 0 |

Client-side, same live worker, same run:

| scenario | n | p50 | p90 | p95 | p99 | max |
|---|---|---|---|---|---|---|
| cold (evict, base ~1.31MB sqlite) | 60 | 190 | 281 | 530 | 875 | 875 |
| warm | 60 | 168 | 190 | 197 | 260 | 260 |

These reproduce v0.4 (180ms p50 / 640ms p99) — same regime; the run's 875ms tail is normal
cold-isolate run-to-run noise at n=60.

**The frozen-clock caveat IS the proof, not just a limitation.** On workerd the wall clock is
frozen inside a synchronous turn. On the SQLite hot path the entire restore (read chunks ->
gunzip -> instantiate QuickJS -> blit linear memory -> re-register host fns) completes inside
**one frozen sync turn**, so server-measured time is exactly 0 even at the tail and even for
real 72s-idle wakes. Therefore the client-observed cold gap (cold p99 875ms vs warm p99 260ms,
~600ms) is spent **outside our code** — Cloudflare cold-isolate spin-up + platform first-WASM
instantiate + WS connect/DO-wake. Confirmed three ways: (a) server phases read 0 at every
percentile; (b) the cold-vs-warm gap exists only client-side; (c) AE recorded `growCount=1` on
all 58 cold restores, proving the B1 pre-size invariant (no blit grow-churn) holds in
production. The one in-kernel cost center that survives is the **R2 overflow path's `readMs`**
for >2MB images, which this base-sized workload did not exercise (R2 reads cross an await, so
they ARE measurable when triggered).

## 4. Bugs / risks, ranked

**Blockers: none.** No DO-kill, no sandbox escape, no durability hole in normal/adversarial use.

**Notable (recovery gaps, not regressions):**
1. **Monotonic-buffer recovery gap (top finding).** A single cell that grows the WASM linear
   buffer to ~256MB (`while(true){a.push(...)}`) is cleanly refused at checkpoint (typed
   `SizeAdmissionError`, socket alive), but its **follow-up eval can hang then WS-1006 the DO,
   and even `reset` 1006s** — session unrecoverable. This is the documented "WASM memory can't
   shrink in place / >45MB" hard limit, now reproduced as a *recovery* gap: the size guard
   catches the dump, but in-cell growth itself isn't bounded and a 256MB-resident isolate is too
   large to reset cleanly. (Note: F4's ~52MB case DID reset cleanly; the gap is at the much
   larger ~256MB resident size.)
2. **Intermittent interrupt-throttle escape.** `while(true){x=1}` trips a clean TimeoutError
   20/20 in isolation, but once WS-1006'd under concurrent fuzzer load — the documented
   load-sensitive workerd interrupt-throttle on property-store loops.

**Minor:**
3. **`host.fetch` to a disallowed host resolves to `{}` instead of rejecting with
   `FetchBlockedError`.** Contained (no network reached, socket alive, default-deny holds), but
   the denial is not observable to the cell as a typed error. Pre-existing v0.3 gap.

**Documented measurement artifacts (not bugs):**
- Server timing doubles (`totalServerMs`/`readMs`/`gunzipMs`/`instantiateMs`) read 0 on the
  SQLite hot path — frozen clock in sync turn. Counts (growCount, nChunks, sizes, usedHeap,
  cell, generation, ok) and the R2-path `readMs` are the always-meaningful doubles.
- AE checkpoint-derived doubles (sizeRaw/usedHeap/store) land as 0 on the bulk of high-burst
  eval rows — reproduced on both canonical and an independent fork, so it is AE adaptive
  sampling/coalescing under burst, not a kernel bug (the DO reply JSON is provably complete).
- Op counts and the per-op taxonomy are unaffected by sampling.

## 5. Merge recommendation + next step

**Recommend MERGE `feat/v0.5` -> `main`.** Observability is live and verified end-to-end (AE SQL
readable, Workers Logs structured). The deep-test pass found **no blockers**: durability is hole-
free under fault injection (11/11), the mutex is provably correct under heavy contention (exact
counters at 200 racers), 0% error / 0 DO-kills at fleet scale, and the cold tail is now
*proven* platform-bound rather than an open question. Consolidate the parallel work
(`chaos/v05-observability` carries chaos + scale + soak harnesses; fuzz on its feat branch)
onto one integration branch first, then merge. Remove the stray artifacts noted in build
gotchas before merge.

**Next deep-test / V1 step:**
1. **Close the monotonic-buffer recovery gap (top priority).** Bound in-cell linear-memory
   growth (hard cap before the isolate reaches an unresettable resident size) so the >45MB
   case degrades to a typed error + clean reset like the ~52MB case, instead of WS-1006.
2. **Fix the `host.fetch` default-deny** to reject with typed `FetchBlockedError` so cells can
   observe the denial.
3. **Exercise + measure the R2 overflow path** (>2MB images) — it is the one in-kernel cost
   center that AE can actually time (crosses an await), and the soak did not hit it.
4. For V1, the cold-start data says further base-path optimization is wasted effort
   (platform-bound); invest instead in the R2 large-image path and in the facet/multi-tenant
   direction.

## Convergence

This section reconciles the five parallel v0.5 work streams (lead BUILD, CHAOS, SCALE, FUZZ,
SOAK+AE) into one agreed picture.

### What all streams agree on
- **Observability is live and verified end-to-end.** AE binding `montydyn_kernel` reads back
  the full per-op taxonomy and a typed `errorName` histogram via the SQL API; Workers Logs emits
  one structured JSON line per op. Confirmed independently by every stream.
- **No DO-kill and no sandbox escape under normal or scaled load.** SCALE (200 racers, 80
  sessions, 40s sustained) and FUZZ (40 sessions, full adversarial battery) both report 0 kills
  / 0% error in normal operation.
- **Durability is hole-free.** CHAOS 11/11 PASS; every crash phase restores to the last
  COMMITTED cell; engine-hash and size-admission guards reject cleanly with the socket alive.
- **The cold-start tail is platform-bound, not kernel-fixable.** SOAK+AE proved server phases
  read 0 at every percentile (the frozen-clock turn IS the proof), `growCount=1` on all cold
  restores (B1 holds in prod), and the ~600ms cold-vs-warm gap exists only client-side.
- **Frozen-clock timing artifact** is consistently understood: counts are authoritative, in-turn
  timings are 0, and only across-await timings (R2 readMs) are meaningful server-side.

### Reconciled discrepancies (honest)
- **Schema drift across streams.** Early reports list doubles
  `[...generation, epoch, committedCell]` (CHAOS/SCALE/FUZZ) while the lead BUILD and SOAK+AE
  list the final shipped doubles `[...generation, gunzipMs, instantiateMs, growCount, nChunks,
  ok]`. The **shipped/authoritative schema is the BUILD/SOAK one** (12 doubles, 7 blobs),
  verified live via SQL readback in section 1. The `epoch`/`committedCell` doubles were an
  earlier iteration; durability assertions used the reply-JSON `committedCell`, so the CHAOS
  conclusions stand regardless.
- **Branch fragmentation.** Streams committed to `chaos/v05-observability` (chaos + scale + soak)
  and `feat/v0.5-observability-scale` (fuzz). The task's target merge is `feat/v0.5 -> main`.
  These must be consolidated onto one integration branch before merge (see step below).
- **DO-kill: only the FUZZ extreme cases kill.** SCALE/CHAOS see 0 kills; FUZZ sees WS-1006 only
  at the ~256MB monotonic-buffer recovery gap and at an intermittent interrupt-throttle escape
  under concurrent load. These are the two real edges and are NOT contradicted by the clean
  scale runs — they live outside the normal envelope.
- **Coordination hygiene noted in BUILD gotchas:** a path mix-up briefly redeployed
  `montydyn-v04` with v05 code (since restored from `v0.4/`), and a stray
  `v0.5/wrangler.jsonc` (dotted) was created in the main repo and removed. A redundant dotted
  `v0.5/` fork built by the SCALE stream was discarded in favor of canonical `v05/`. Verify
  these are clean before merge.

### Most valuable single output
The **reproducible DO-kill** at the monotonic-buffer recovery gap (FUZZ finding 1): a cell that
grows the WASM linear buffer to ~256MB is cleanly refused at checkpoint, but the follow-up eval
hangs -> WS-1006 and even `reset` 1006s, making the session unrecoverable. This is the one
concrete, reproducible failure worth fixing first. It is a recovery gap on a documented v0.2
hard limit (WASM can't shrink in place), not a new v0.5 regression, and does NOT compromise
committed durability (no torn/lost snapshot) — but it is the clearest V1 blocker candidate.

### Converged recommendation
**MERGE after consolidation.** Steps: (1) reconcile to the shipped 12-double/7-blob schema;
(2) consolidate chaos + scale + fuzz + soak harnesses onto one `feat/v0.5` branch; (3) confirm
`montydyn-v04`/`v0.4/` and the stray dotted dirs are clean; (4) merge `feat/v0.5 -> main`.
Then V1: close the ~256MB recovery gap (bound in-cell linear-memory growth), fix `host.fetch`
default-deny to a typed `FetchBlockedError`, and instrument/measure the R2 large-image overflow
path (the one server-timeable in-kernel cost center).
