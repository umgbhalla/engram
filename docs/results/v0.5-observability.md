# montydyn v0.5 — Cloudflare server-side observability (Analytics Engine + Workers Logs)

> Built on v0.4 (`v0.5/`), deployed **`montydyn-v05`**
> (https://montydyn-v05.umg-bhalla88.workers.dev). Branch `feat/v0.5`. R2 keys namespaced
> `v05/`. v0/v0.1/v0.2/v0.3/v0.4 + their workers + the facet spike all UNTOUCHED. 2026-06-01.

## Goal

Instrument the kernel with Cloudflare **server-side** observability so every op is
measured at the source (no client/network noise, no coordinated omission):

- **Analytics Engine (AE)** datapoint on EVERY op, with a rich blobs/doubles schema,
  queryable via the AE SQL API.
- **Workers Logs**: a structured JSON console line per op (`observability.enabled` kept on),
  queryable in the dashboard / `wrangler tail`.

## What changed vs v0.4 (additive only)

- `wrangler.jsonc`: added the `analytics_engine_datasets` binding
  `{ binding: "AE", dataset: "montydyn_kernel" }`. `observability.enabled` kept `true`.
  Worker renamed `montydyn-v05`; R2 prefix `v04/` -> `v05/`.
- `src/lib.rs`: a `Datapoint` struct + `emit()` helper. `emit()` (a) `console.log`s one
  structured JSON line (`{"ev":"op", ...}`) and (b) calls `env.AE.writeDataPoint({indexes,
  blobs, doubles})` via JS interop (`Reflect` on the `Env` JsValue — the worker crate has no
  first-class AE binding). Every op handler calls `emit()`.
- No glue.js / engine changes. The instrumentation is a pure no-op if the AE binding is
  absent (so the same code runs unmodified on a worker without the dataset).

## Ops instrumented (the `op` enum — AE `blob1`)

| op | when |
|---|---|
| `create` | `{t:"create"}` that established a fresh kernel |
| `restore` | a cold/lazy restore (emitted by both `create` and `eval` when `ensure_glue()` restored from a snapshot) |
| `eval` | a successful cell eval |
| `error` | a cell whose top-level threw (non-Timeout) — `errorName` carries the JS error name |
| `timeout` | a cell preempted by the interrupt-tick budget (`errorName="TimeoutError"`) |
| `fetch` | a cell whose source used `host.fetch` (best-effort egress marker; `errorName` carries FetchBlockedError/FetchError) |
| `size-reject` | `dump()` refused the snapshot (`SizeAdmissionError` / over the dump buffer ceiling) — checkpoint failed but socket stays alive |
| `checkpoint` | a non-size checkpoint failure |
| `gen` | liveness query (lazy — no instantiate) |
| `ping` | cheap liveness ping (lazy) |
| `evict` | in-memory kernel dropped (test hook); `valueType` = `dropped`/`noop` |
| `reset` | snapshot cleared + epoch bumped |

## AE datapoint schema (stable)

`writeDataPoint({ indexes, blobs, doubles })`

### indexes (sampling/partition key; max 1, <=96 bytes)
| idx | field | meaning |
|---|---|---|
| `index1` | `doId` | the Durable Object id (the session) |

### blobs (max 20; <=5120 bytes total) — AE SQL columns `blob1..blob7`
| col | field | meaning |
|---|---|---|
| `blob1` | `op` | the op enum above |
| `blob2` | `restoreSource` | `fresh` / `warm` / `sqlite-restore` / `r2-restore` |
| `blob3` | `store` | snapshot store for this op's checkpoint: `sqlite` / `r2` / `` |
| `blob4` | `errorName` | JS error name on error/timeout/fetch/size-reject (`` otherwise) |
| `blob5` | `configClock` | session clock mode (`seeded` / `real`) — set on create/restore |
| `blob6` | `valueType` | eval result type (`number`/`object`/`error`/...) or evict `dropped`/`noop` |
| `blob7` | `label` | reserved (currently `` ) |

### doubles — AE SQL columns `double1..double12`
| col | field | meaning |
|---|---|---|
| `double1` | `totalServerMs` | server-side restore span (whole `ensure_glue()`), 0 when warm/no-restore |
| `double2` | `readMs` | snapshot byte-fetch time (SQLite read ~0 frozen; R2 `get` measurable) |
| `double3` | `sizeRaw` | raw serialized snapshot bytes (checkpoint) |
| `double4` | `sizeGz` | gzipped snapshot bytes (checkpoint) |
| `double5` | `usedHeap` | QuickJS live used heap at checkpoint |
| `double6` | `cell` | committed cell index |
| `double7` | `generation` | DO generation counter (cold-wake evidence) |
| `double8` | `gunzipMs` | glue gunzip phase (across the gunzip await; ~0 sub-await) |
| `double9` | `instantiateMs` | glue instantiate+grow+blit phase (~0 sub-await) |
| `double10` | `growCount` | exact `WebAssembly.Memory.grow` calls during restore (B1: should be <=1) |
| `double11` | `nChunks` | SQLite chunks written for this checkpoint |
| `double12` | `ok` | 1.0 success / 0.0 failure |

> **Frozen-clock caveat (inherited from v0.4):** workerd freezes the wall clock within a
> synchronous turn (spectre mitigation). So sub-turn timings (`gunzipMs`, `instantiateMs`,
> SQLite `readMs`) read `0` — that is honest "sub-await", not "free". The doubles that ARE
> always meaningful are the **counts** (`growCount`, `nChunks`, sizes, `usedHeap`, `cell`,
> `generation`, `ok`) and `readMs`/`totalServerMs` when an await actually yields (R2 path).

## Workers Logs

Each op also emits a single structured JSON line via `console.log`:

```json
{"ev":"op","op":"eval","doId":"...","restoreSource":"warm","store":"sqlite",
 "errorName":"","configClock":"","valueType":"number","totalServerMs":0,"readMs":0,
 "sizeRaw":1310748,"sizeGz":108718,"usedHeap":...,"cell":0,"generation":1,
 "gunzipMs":0,"instantiateMs":0,"growCount":0,"nChunks":2,"ok":true}
```

Query in `wrangler tail montydyn-v05 --format=pretty` or the dashboard Logs view; filter on
`$.op`, `$.errorName`, `$.doId`.

## How to query AE via the SQL API

```bash
set -a && . .env && set +a   # CLOUDFLARE_ACCOUNT_ID + CLOUDFLARE_API_TOKEN
ACC="$CLOUDFLARE_ACCOUNT_ID"

curl -s -X POST \
  "https://api.cloudflare.com/client/v4/accounts/$ACC/analytics_engine/sql" \
  -H "Authorization: Bearer $CLOUDFLARE_API_TOKEN" \
  --data "SELECT blob1 AS op, count() AS n, avg(double1) AS avg_total_ms
          FROM montydyn_kernel
          WHERE timestamp > NOW() - INTERVAL '1' HOUR
          GROUP BY op ORDER BY n DESC FORMAT JSON"
```

Useful queries:

```sql
-- error/timeout rate by name
SELECT blob1 AS op, blob4 AS errorName, count() AS n
FROM montydyn_kernel WHERE timestamp > NOW() - INTERVAL '1' HOUR
GROUP BY op, errorName ORDER BY op;

-- cold-restore latency distribution (R2 path readMs is the real cost)
SELECT blob2 AS source,
       quantileWeighted(0.50)(double1, 1) AS p50_ms,
       quantileWeighted(0.95)(double1, 1) AS p95_ms,
       avg(double2) AS avg_readMs
FROM montydyn_kernel WHERE blob1 = 'restore'
  AND timestamp > NOW() - INTERVAL '1' DAY GROUP BY source;

-- snapshot size growth per session
SELECT index1 AS doId, max(double4) AS max_gz, max(double6) AS max_cell, max(double3) AS max_raw
FROM montydyn_kernel WHERE blob1 = 'eval'
  AND timestamp > NOW() - INTERVAL '1' DAY GROUP BY doId ORDER BY max_gz DESC;

-- B1 invariant audit: grow churn during restore should be <=1
SELECT max(double10) AS max_grow_count, count() AS restores
FROM montydyn_kernel WHERE blob1 = 'restore' AND timestamp > NOW() - INTERVAL '1' DAY;
```

## SMOKE — live verification (montydyn-v05)

`v0.5/obs-smoke.mjs` + `v0.5/obs-errs.mjs` drove a session through every op type
(create, ping, gen, eval, error, timeout, fetch, fetch-blocked, evict, cold-restore eval,
reset). State survived the evict (`x` restored on cold eval, cell 6). Then the AE SQL API
was queried and the datapoints came back:

```
op         errorName     n
create                   2
error      TypeError     1
eval                     2
evict                    1
fetch                    1
gen                      1
ping                     1
reset                    1
restore                  1
timeout    TimeoutError  1
```

(`store='sqlite'` confirmed on eval datapoints; `ok=1` success / `ok=0` on error+timeout.)
All 10 distinct op types are queryable with correct blob/double attribution. **AE queryable
= TRUE.**

### Gotchas observed
- **AE ingestion lag:** datapoints take ~30–90s to become queryable; a query immediately
  after the ops returns `rows: 0` (schema present, data empty). Re-query after a minute.
- **AE sampling:** under load AE samples; use `count()` (which auto-reweights via the
  internal `_sample_interval`) rather than assuming 1 row == 1 op at high volume. At smoke
  volume counts are exact.
- The worker `montydyn-v04` was briefly redeployed with v05 code during a path mix-up, then
  **redeployed from `v0.4/` to restore its original code** — v04 is back to its real build.
