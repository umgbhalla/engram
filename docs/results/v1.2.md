# montydyn v1.2 — Verdict

**VERDICT: GO** (merge). All three test areas PASS live against `https://montydyn-v12.umg-bhalla88.workers.dev`. Zero regression vs v1.1.

## What shipped
Per-tenant API-key auth + Analytics Engine (AE) metering + `/usage` aggregation, turning montydyn into a billable multi-tenant SaaS seam.

- **Auth**: tenant registry in supervisor control-plane SQLite (`tenantId -> SHA-256-hashed key, created, plan, revoked`); raw key shown once at mint. Authoritative registry on a canonical `sup-auth` shard; the Worker entry resolves every data-plane key against it so auth works from any of the 64 session shards.
- **Data-plane gate**: no/invalid/revoked key -> 401. Resolved tenant injected as `x-md-tenant` header so the client cannot spoof tenant scope; `facetName` is keyed by tenant for hard isolation.
- **Admin**: mint/list/revoke routes gated by `ADMIN_TOKEN` secret (POST/GET/DELETE `/admin/keys`). List returns hashes/metadata only.
- **Metering**: per-request `writeDataPoint` (blob1=tenant, blob2=metric, blob3=session, double1=value) on eval, checkpoint bytes, keep-warm heartbeats, per-tenant facet counts.
- **Usage**: `GET /usage?tenant=&window=` queries AE SQL for per-tenant aggregates (evals, bytes, warmSeconds, facetPeak, activeSessions). Admin sees all; tenant sees own via its key.

## Known limitations / handoff
- AE ingestion lag ~60-90s before `/usage` reflects events.
- AE SQL rejects `COUNT(*)` (use `count()`); `uniq()` unsupported (`count(DISTINCT ...)` works). Two such bugs fixed mid-smoke (commit 75fe456).
- `montydyn_kernel` AE dataset is shared across v0.5/v1.1/v1.2 with differing blob schemas. `blob2 IN (eval,bytes,warm,facet)` filter removes most stale rows; a few coincidental old rows still surface as zero-metric tenants in the admin-all `/usage` view (harmless; tenant self-scoped view is clean). A dedicated v1.2 dataset would fully fix.
- `URL.searchParams` undefined — pre-existing v1.1 engine limitation, NOT a v1.2 regression.
- **Action for lead**: tester reset `ADMIN_TOKEN` via `wrangler secret put` to a known test value (current value in `/tmp/md_admin_tok`). Rotate `ADMIN_TOKEN` back to the intended production value.
- Reused R2 `montydyn-snapshots` + AE `montydyn_kernel`; no resources deleted.

## Is montydyn a billable multi-tenant SaaS seam?
Yes. Authentication is enforced and unspoofable, tenant isolation is hard (per-tenant facet/SQLite), and metering produces exact, per-tenant, self-scoped usage aggregates suitable for billing. The remaining gap to production billing is a dedicated AE dataset (to drop schema-shared stale rows) and longer-window `warmSeconds` validation.
