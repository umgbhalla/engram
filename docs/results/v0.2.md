## GATE-FIX PASS (this update) — two independent-gate failures closed

The first V0.2 pass shipped the dump-side used-heap admission but left **two live gate
failures**. Both are now fixed and verified against `montydyn-v02` (52/52 smoke):

### P0 — COLD-RESTORE WEDGE (fixed)
**Bug:** the dump-side guard was correctly on used-heap, but the **RESTORE** guard
(`glue.js`) still rejected on the **raw image bytes** (`serialized.byteLength > MAX_RAW_BYTES`
= 20MB). Because WASM linear memory is monotonic, a session that ever spiked its buffer past
~18MB raw, then freed + checkpointed (fine warm — small used heap), could **NEVER cold-restore**:
every restore threw `SizeAdmissionError`; the only escape was `reset` (total state loss).
**Fix:** the restore guard now admits on the snapshot's **recorded `usedHeap`** (persisted in
the manifest at dump time — new `used_heap` column, wired through `lib.rs` → `restore()`), not
the raw bytes. A separate **safe-to-instantiate raw ceiling** (`MAX_RESTORE_RAW_BYTES` = 45MB,
pinned to the dump-side `MAX_DUMP_BUFFER_BYTES`) is kept ONLY so a genuinely-too-big image still
fails safe — if an image could be dumped it can be restored. A freed-then-small session whose
raw high-water is high but whose used heap is tiny now **MUST restore**, and does.
- Live evidence (smoke `P0w`): a **24MB spike → raw image 26.5MB** (the OLD guard rejected this)
  → free → evict → **cold restore SUCCEEDS** (`ok=true value=777 restoreSource=sqlite-restore`,
  no `SizeAdmissionError`). The exact wedge, now closed.

### sizeGz typed-array measurement — investigated, NO code bug
The gate flagged a 24MB incompressible LCG array reporting `gz≈0.207MB` (impossible). Root cause
is **the gate's test fill, not montydyn**: the probe LCG used `s = s*1103515245 + 12345`, whose
multiply overflows JS double precision (`1103515245 × 2^32 ≈ 4.7e18 > 2^53`), degenerating to
**~89% zero bytes** — highly compressible, hence the tiny gz. The dump captures the **full WASM
linear memory** (`vm.snapshot()` → `new Uint8Array(memory.buffer).slice()`, verified at
`quickjs-wasi/dist/index.js:1482`) and gzips it with `CompressionStream("gzip")`. Verified with a
**precision-safe 32-bit LCG (`Math.imul`)**: a genuinely incompressible 24MB array serializes to
26.48MB raw and gzips to **25.27MB** (no compression) — node `zlib` agrees byte-for-byte. So
`sizeRaw`/`sizeGz`/store-routing are **correct**; nothing under-captures. (Documented; smoke `P0w`
asserts the raw image is the full >20MB image.)

### P1 — GLOBAL-WRITE LOOP ESCAPED PREEMPTION (fixed)
**Bug:** empty `while(true){}` and local-write loops tripped a typed `TimeoutError`, but
`while(true){x=1}` / `while(true){globalThis.x=1}` (per-iteration global/object property write)
did **not** — they rode to the CF DO wall limit → WS 1006 (socket dead).
**Root cause (found by live probing on workerd):** the interrupt handler IS invoked for every
loop shape (a 1-tick budget trips all of them), and it already decremented the budget on every
invocation. The real culprit is a **workerd-specific throttle: workerd stops calling the host
interrupt callback after ~1.6k invocations within a single synchronous turn.** A property-store
loop accrues interrupt ticks at the *same* cadence as an empty loop, so with the v0.2 default of
8000 ticks the budget sat **above** that throttle cap → the handler stopped firing before the
budget was reached → the loop rode to the wall. (Wall-clock can't save it: `Date.now()` is frozen
during sync execution on workerd — re-confirmed: a 500ms wall budget with a huge tick budget never
trips.)
**Fix:** the interrupt tick budget is hard-capped **below** the workerd throttle cap — **default
1200 ticks, max 1500** (`normalizeConfig`) — and the handler is kept as cheap as possible (a single
closure-local decrement; no `Date.now()`/`this` reads on the hot path) to maximize invocations
before throttling. At this budget EVERY infinite-loop shape trips a typed `TimeoutError` in <0.7s
with the socket alive.
- Live evidence (smoke `P1`, each on a fresh DO/kernel — the throttle is sensitive to prior
  in-kernel work): empty `while(true){}` → TimeoutError 238ms; `while(true){x=1}` → 686ms;
  `while(true){globalThis.x=1}` → 480ms; all `readyState=1`, next eval `=42`. A legit **6M-iter**
  loop completes (`ok=true`).
- **Documented workerd limit (NOT a regression):** because the throttle cap (~1.6k invocations)
  sits *just below* what a **10M** tight modulo loop needs (~1.7k ticks at this cadence), a 10M
  tight loop on workerd cannot both complete AND leave room to trap `while(true){x=1}` — there is
  no single budget satisfying both. We prioritize the gate-critical preemption (every infinite loop
  trips, socket alive) and certify a legit **~6M-iteration** loop as the in-envelope "heavy loop"
  benchmark (completes ~0.5s at the 1500 cap). Users needing larger CPU-bound bursts should chunk
  across cells (each cell re-arms the budget) — the same pattern the `P0w` 24MB fill uses.

## 1. P0 — BUG-2/4 memory reclaim

### Root cause (confirmed empirically)
WASM linear memory is **monotonic**: `memory.grow` never shrinks the buffer in place, and QuickJS's dlmalloc does not compact live allocations downward. So `snapshot()` (which copies the entire `memory.buffer`) always stays at the high-water mark. V0.1's size-guard measured `memory.buffer.byteLength`, which never drops → a spike-then-free permanently wedged checkpointing.

Local probe (`quickjs-wasi` 3.0.0, this wasm):
```
after-alloc 40MB incompressible: used=42.0MB  buf=43.2MB  gz=1.61MB
after-free+gc:                    used= 0.1MB  buf=43.2MB  gz=1.61MB   <- buf+gz pinned
restore-into-fresh:               buf=43.2MB                            <- re-grows to old size
after-free last-non-zero byte:    at 43.2MB of 43.2MB                   <- freed region NOT zeroed
```

### Fixes
1. **Used-heap admission guard (the un-wedge).** `dump()` admits on `max(memoryUsedSize, mallocSize)`. After free+GC this drops to baseline (~75KB live in tests), so checkpointing is admitted again.
2. **Arena scrub (snapshot/store shrink).** After a free, `dump()` allocates zero-initialized `ArrayBuffer`s across the freed slack inside the VM, then frees + GC. dlmalloc re-serves the just-freed chunks (spec-zeroed), so the freed pages are now zero and gzip collapses them. The scrub is **bounded** (≤24MB, ≈0.6× of the slack) so it never grows memory, and is **skipped above a 32MB buffer ceiling** for safety.
   - Local: 30MB free → gz **0.84MB → 0.18MB**, namespace intact.
3. **Fail-safe dump ceiling.** `snapshot()`+serialize copy the full buffer ~2-3x and OOM is uncatchable (Error 1102 / WS 1006 / hang). Once the buffer exceeds ~45MB, `dump()` throws a typed `SizeAdmissionError` **before** touching the full buffer (before `runGC`, before `snapshot`) — socket alive, prior snapshot intact, `reset` recovers.

### What shrinks vs what doesn't
| Quantity | After a free | Mechanism |
|---|---|---|
| **Used heap** (admission figure) | **shrinks to baseline** | `getMemoryUsage()` post-GC |
| **gz image / stored bytes** | **shrinks back** (store r2→sqlite) | arena scrub zeroes freed pages |
| **Raw `memory.buffer`** | does **NOT** shrink | WASM monotonic; dlmalloc no downward compaction |

True compaction into a smaller fresh instance is **not feasible** without value-level re-serialization (which loses pending-promise/closure fidelity): restoring a snapshot re-grows memory to the old buffer size (verified), and the live high-water after free+GC+scrub remains at the top of the buffer (dlmalloc does not move live chunks — verified via truncation probe: even after scrub the last non-zero byte was at the buffer top, so a truncated snapshot would lose live state). We therefore ship fix #1 (un-wedge) + the scrub (gz shrink) + the fail-safe ceiling, and document the raw-buffer limit.

### Live evidence (smoke)
```
P0 baseline:        usedHeap=0.08MB gz=0.11MB store=sqlite
P0 after-40MB-alloc: ok=true used=42.0MB gz=0.34MB store=sqlite      (in-envelope spike checkpoints)
P0 after-free:       usedHeap=0.08MB gz=0.34MB store=sqlite          (CHECKPOINTS AGAIN; used reclaimed 42MB->75KB)
P0c 60MB live:       ok=false typed=true (SizeAdmissionError: buffer 64225280 > 47185920 ... reset to recover)
```
- `P0 free cell CHECKPOINTS AGAIN (no permanent SizeAdmissionError)` — **PASS** (the exact v0.1 wedge, now fixed).
- `P0 used heap reclaimed after free (<2MB)` = 75059 — **PASS**.
- `P0 store returns to sqlite after free` — **PASS**.
- `P0c >ceiling spike refused with typed error (no silent wedge)` — **PASS** (socket alive, no hang/1006).

## 2. P1 — BUG-3 reliable preemption (incl. the global-write escape)

### Root cause (corrected by the gate-fix pass — see top of doc for the full narrative)
Two things compound on workerd: (1) `Date.now()` is **frozen** during synchronous execution, so a
wall-clock deadline never trips; and (2) **workerd stops invoking the host interrupt callback after
~1.6k invocations within a single synchronous turn.** The interrupt fires for every loop shape (a
1-tick budget trips all of them) and the handler already decremented on every invocation — but the
first V0.2 pass set the budget to **8000**, which sat ABOVE the ~1.6k throttle cap, so a tight loop
(especially a property-store `while(true){x=1}`) never reached the budget before workerd stopped
calling the handler → it rode to the ~30s DO wall limit → WS 1006.

### Fix
Hard-cap the tick budget **below** the workerd throttle: **default 1200, max 1500 ticks**
(`normalizeConfig`). Keep the handler maximally cheap (single closure-local decrement; no
`Date.now()`/`this` reads on the hot path) so as many invocations as possible land before the
throttle. The wall-clock check remains a best-effort secondary (frozen on workerd, useful on
node/tests).

### Live evidence (smoke, each loop on a FRESH DO/kernel)
```
P1 empty while(true){}            -> TimeoutError in 238ms  readyState=1  next eval=42
P1 while(true){x=1}               -> TimeoutError in 686ms  readyState=1  next eval=42
P1 while(true){globalThis.x=1}    -> TimeoutError in 480ms  readyState=1  next eval=42
P1 legit 6M-iter loop completes (not tripped)  ok=true
```
All **PASS**. Every infinite-loop shape returns `{ok:false, error:TimeoutError}` with the socket
open (no WS 1006) and the next eval works. **Documented workerd limit:** a 10M *tight* loop cannot
both complete and leave headroom to trap `while(true){x=1}` (the throttle cap ~1.6k sits just below
the 10M loop's ~1.7k-tick need); the certified heavy benchmark is **~6M iters** at the 1500 cap, or
chunk a larger burst across cells (each cell re-arms the budget).

## 5. Remaining gaps / next

- **Raw buffer never shrinks in place** (documented limit). A >~45MB buffer session must `reset` to reclaim; the dump fails safe. For workloads needing post-spike durability beyond the envelope, the path is value-level re-serialization (fidelity trade-off) or a smaller per-session memory cap — deferred.
- **P3 (unchanged from V0.1):** `fetch` allowlist plumbed but inert (no egress); error-*value* preview drops message/stack.
- **Scrub on workerd**: validated indirectly (40MB case did not need it — sparse data already compresses; 30MB local case shows the shrink). The scrub is gated under a 32MB buffer ceiling and never grows memory, so it cannot OOM.
