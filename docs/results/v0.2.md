## TL;DR

V0.2 lands the two highest-value open bugs from the V0.1 verification, plus P2:

- **P0 (BUG-2/4) memory reclaim — FIXED (un-wedge) + image shrink, with a documented hard limit.** The size-admission guard now measures QuickJS's **actual used heap** (`getMemoryUsage().memoryUsedSize`), not the monotonic `memory.buffer`. A session that spikes **within the operating envelope** then frees **can checkpoint again** (no permanent `SizeAdmissionError`), and an **arena scrub** zeroes the freed pages so the gz/stored image shrinks back (store returns to sqlite). The **raw buffer does NOT shrink in place** — a fundamental WASM constraint; a >~45MB buffer fails the dump *safe* (typed error, socket alive) instead of OOM-crashing.
- **P1 (BUG-3) reliable preemption — FIXED.** The interrupt-tick budget is now the hard primary; an empty `while(true){}` trips a typed `TimeoutError` in **<1s** (default 8000 ticks), socket alive, next eval works. A 10M-iteration legit loop still completes.
- **P2 host-tool kv state — DONE.** `host.kv.*` data survives a cold restore (serialized in the snapshot manifest, re-hydrated on restore).

No regressions: BUG-1 error handling, dynamic config + tools across evict, seeded clock/RNG, state survival, atomic checkpoint all green.

## 1. P0 — BUG-2/4 memory reclaim

### Root cause (confirmed empirically)
WASM linear memory is **monotonic**: `memory.grow` never shrinks the buffer in place, and QuickJS's dlmalloc does not compact live allocations downward. So `snapshot()` (which copies the entire `memory.buffer`) always stays at the high-water mark. V0.1's size-guard measured `memory.buffer.byteLength`, which never drops → a spike-then-free permanently wedged checkpointing.

Local probe (`quickjs-wasi` 3.0.0, this wasm):
```
after-alloc 40MB incompressible: used=42.0MB  buf=43.2MB  gz=1.61MB
after-free+gc:                    used= 0.1MB  buf=43.2MB  gz=1.61MB   <- buf+gz pinned
restore-into-fresh:               buf=43.2MB                            <- re-grows to old size
after-free last-non-zero byte:    at 43.2MB of 43.2MB                   <- freed region NOT zeroed
```

### Fixes
1. **Used-heap admission guard (the un-wedge).** `dump()` admits on `max(memoryUsedSize, mallocSize)`. After free+GC this drops to baseline (~75KB live in tests), so checkpointing is admitted again.
2. **Arena scrub (snapshot/store shrink).** After a free, `dump()` allocates zero-initialized `ArrayBuffer`s across the freed slack inside the VM, then frees + GC. dlmalloc re-serves the just-freed chunks (spec-zeroed), so the freed pages are now zero and gzip collapses them. The scrub is **bounded** (≤24MB, ≈0.6× of the slack) so it never grows memory, and is **skipped above a 32MB buffer ceiling** for safety.
   - Local: 30MB free → gz **0.84MB → 0.18MB**, namespace intact.
3. **Fail-safe dump ceiling.** `snapshot()`+serialize copy the full buffer ~2-3x and OOM is uncatchable (Error 1102 / WS 1006 / hang). Once the buffer exceeds ~45MB, `dump()` throws a typed `SizeAdmissionError` **before** touching the full buffer (before `runGC`, before `snapshot`) — socket alive, prior snapshot intact, `reset` recovers.

### What shrinks vs what doesn't
| Quantity | After a free | Mechanism |
|---|---|---|
| **Used heap** (admission figure) | **shrinks to baseline** | `getMemoryUsage()` post-GC |
| **gz image / stored bytes** | **shrinks back** (store r2→sqlite) | arena scrub zeroes freed pages |
| **Raw `memory.buffer`** | does **NOT** shrink | WASM monotonic; dlmalloc no downward compaction |

True compaction into a smaller fresh instance is **not feasible** without value-level re-serialization (which loses pending-promise/closure fidelity): restoring a snapshot re-grows memory to the old buffer size (verified), and the live high-water after free+GC+scrub remains at the top of the buffer (dlmalloc does not move live chunks — verified via truncation probe: even after scrub the last non-zero byte was at the buffer top, so a truncated snapshot would lose live state). We therefore ship fix #1 (un-wedge) + the scrub (gz shrink) + the fail-safe ceiling, and document the raw-buffer limit.

### Live evidence (smoke)
```

P0 after-40MB-alloc: ok=true used=42.0MB gz=0.34MB store=sqlite      (in-envelope spike checkpoints)
P0 after-free:       usedHeap=0.08MB gz=0.34MB store=sqlite          (CHECKPOINTS AGAIN; used reclaimed 42MB->75KB)
P0c 60MB live:       ok=false typed=true (SizeAdmissionError: buffer 64225280 > 47185920 ... reset to recover)
```
- `P0 free cell CHECKPOINTS AGAIN (no permanent SizeAdmissionError)` — **PASS** (the exact v0.1 wedge, now fixed).
- `P0 used heap reclaimed after free (<2MB)` = 75059 — **PASS**.

## 2. P1 — BUG-3 reliable preemption

### Root cause
On workerd `Date.now()` is **frozen** during synchronous execution, so a wall-clock deadline never trips. V0.1 used a 30000-tick budget but the handler's wall-clock branch was load-bearing on hosts where time advances; on workerd 30000 ticks (~20k ops/tick ≈ 600M ops) could ride past the ~30s DO wall limit → WS 1006 for empty/CPU-bound loops.

### Fix
The interrupt handler decrements a **hard tick budget on every invocation** (the primary), so ANY loop — empty `while(true){}` included — hits the budget. The wall-clock check is a secondary guard only. Default lowered **30000 → 8000 ticks**. Calibration (local, ~20k ops/tick): a 10M-iteration loop ≈ 2000 ticks (completes); an empty loop accrues ticks at ~20k iters/tick and trips at 8000 ticks well under the DO wall limit.

### Live evidence (smoke)
```
P1 empty while(true){} returned in 375ms
P1 empty while(true){} -> TimeoutError   {"name":"TimeoutError","message":"cell exceeded execution budget (8000 ticks / 5000ms wall)"}
P1 socket still OPEN after empty loop (no WS 1006)  readyState=1
P1 next eval works after empty-loop timeout  value=25
P1 heavy 10M-iter loop completes (not tripped)  ok=true value=3
```
All **PASS**. An empty `while(true){}` now returns `{ok:false, error:TimeoutError}` with the socket open (readyState 1, no WS 1006), the next eval works, and a legit 10M-iter loop completes.

## 3. P2 — host-tool kv state persistence

`host.kv.*` state lives outside WASM linear memory. V0.2 serializes the kv Map in `dump()` (`kvJson`), persists it in the snapshot manifest (`kv_json` column, travels through SQLite and R2 paths), and re-hydrates it on `restore()` after `_applyConfig` (preserving the Map identity the host tools close over).

Live evidence (smoke): `kv.get('alpha')` = 100 and `kv.keys()` = `["alpha","beta"]` **survive an evict + cold restore** (`restoreSource=sqlite-restore`). Both **PASS**.

## 5. Remaining gaps / next

- **Raw buffer never shrinks in place** (documented limit). A >~45MB buffer session must `reset` to reclaim; the dump fails safe. For workloads needing post-spike durability beyond the envelope, the path is value-level re-serialization (fidelity trade-off) or a smaller per-session memory cap — deferred.
- **P3 (unchanged from V0.1):** `fetch` allowlist plumbed but inert (no egress); error-*value* preview drops message/stack.
- **Scrub on workerd**: validated indirectly (40MB case did not need it — sparse data already compresses; 30MB local case shows the shrink). The scrub is gated under a 32MB buffer ceiling and never grows memory, so it cannot OOM.

### Verdict
V0.2 is a strict, no-regression hardening over V0.1. The P0 size-trip wedge (the #1 durability/cost liability) is fixed for in-envelope workloads with the raw-buffer limit honestly documented; P1 preemption is now reliable for all loop shapes; P2 kv persists. **Do not merge** (per task) — left on `feat/v0.2` for review.
